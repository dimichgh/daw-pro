# Design Language — "Glass Cockpit"

Dark, glassy, information-dense but calm. Every glow means something. Think: night-flight avionics meets high-end studio hardware, rendered with macOS-native polish.

## Foundations

- **Surfaces**: near-black layered glass. Base `#0B0D12`, raised panels `#12151D` with 1px `white @ 6%` hairline borders and subtle inner shadow. Use translucency (`.ultraThinMaterial`-style) sparingly, only for overlays.
- **Text**: SF Pro for labels; **SF Mono for every numeric readout** (time, tempo, dB, Hz). Readouts are "digital": mono font, slight letterspacing, colored glow. Text runs a measured hierarchy — `textPrimary` (primary content) › `textSecondary` (legible secondary labels, ≥ 4.5:1) › `textDim` (muted captions/micro-labels, ≥ 4.5:1) › `textFaint` (placeholder & decorative floor, ≥ 3.0:1); every value is measured against the raised-panel surface. Never dim a real label below 4.5:1 with ad-hoc `.opacity()` — reach for the right token.
- **Accent system** (semantic, not decorative):
  - Cyan `#3EE6FF` — playback/position/active state
  - Amber `#FFB84D` — record/warning/clipping-adjacent
  - Red `#FF4D5E` — clip/overload/destructive
  - Green `#5DFF9F` — healthy signal, success
  - Violet `#B48CFF` — AI/generated content (everything AI-touched carries violet)
- **Glow recipe**: color at full opacity core + same-color shadow (radius 6–10, opacity 0.5–0.7) + faint wide bloom (radius 20, opacity 0.15). Glow intensity may encode magnitude (meter brightness follows level). Never glow static chrome.

## Components

- **Meters**: segmented LED-style bars (Canvas-drawn), green→amber→red zones, peak-hold tick that decays, glow tracks level. 60fps via `TimelineView`.
- **Transport**: oversized readout cluster — bars.beats + SMPTE time in mono cyan, tempo editable inline, glowing play/record buttons with state halos.
- **Knobs/faders**: flat dark caps, thin neon value arcs/tracks; value readout appears on touch. Double-click resets. Fine-drag with ⌥.
- **Waveforms**: dual-tone (body + brighter RMS core), violet tint when clip is AI-generated.
- **Panels**: every panel has Simple and Pro density modes; Simple shows the 20% of controls used 80% of the time with plain-language labels ("Punchier" over "Ratio 4:1" — Pro mode shows both). Mode is a themed segmented chip pair (SIMPLE / PRO), the active half cyan-lit; Simple is the default. **Density is a per-panel, app-sticky PREFERENCE, never project data** — each panel owns its own mode (a beginner may want a Pro piano roll but a Simple mixer), it survives close/reopen and app relaunch (persisted as `panelDensity.<panelID>` in UserDefaults), and it is NEVER written into the project file. The chip pair is one shared component — `DAWApp/Components/SimpleProToggle`, bound to a `DAWAppKit.PanelDensityStore` + a stable panel ID — so it is pixel-identical everywhere (SF Mono 9 pt bold uppercase, tracking 0.6, active half `playback` on an 0.18 fill, on a raised chip with a hairline border). The headless model lives in `DAWAppKit.PanelDensity`/`PanelDensityStore` (injected persistence backing, `@Observable`); `debug.panelDensity {panel, mode}` stages a mode for captures/E2E (debug tier — not an MCP tool). The per-panel Simple/Pro plan lives in `docs/research/simple-pro-inventory.md`. Panels that already pass the beginner test adopt the model but render identically in both modes (their modes coincide); the piano roll is the reference implementation. As of sp-d, **density coverage is uniform**: exactly four surfaces carry live chips (piano roll, mixer, arrange, transport); every other panel's Simple and Pro modes **COINCIDE by design and must never grow a do-nothing toggle** — a toggle that changes nothing is worse than none (the per-surface roll-up lives in `docs/research/simple-pro-inventory.md` §"Density adoption status").
- **Arrange timeline**: fixed pixels-per-beat (16 pt/beat, no zoom yet); one lane per track in sidebar order. Clips are rounded tinted blocks — signal-green for audio, playback-cyan for MIDI, **violet whenever AI-touched** — with a name label when width allows, selected clip brighter + glowing. Beat grid is hairlines; bar lines brighter; bar numbers in SF Mono. Playhead is a glowing cyan hairline driven by transport position (offset a view, never a per-frame Canvas redraw). A track's header carries an automation disclosure (axis-chart glyph, cyan-lit + glowing when the track has an enabled non-empty lane); opening it grows a beat-aligned breakpoint-editor row under the clip lane and a matching control panel in the sidebar (so both columns stay aligned). The editor is a Canvas neon polyline (glow recipe: wide faint bloom under a bright core) with glowing breakpoint dots and a faint dashed neutral guide (unity gain / center pan); **volume lanes track the fader's cyan, pan lanes stay neutral white** (mixer semantics — pan claims no accent), a **disabled lane renders dimmed** (its curve is inert; the fader/knob is manual). Click empty space adds a point, drag moves it with a live SF Mono value/beat readout by the cursor (dB for volume, L/C/R for pan), double-click or select+⌫ deletes; edits redraw only on interaction (no TimelineView). Target picker is volume/pan in v0 (a themed VOL/PAN chip pair, the active half cyan-lit, a dot marking which already has a lane); an enable chip (green ON / dim OFF) flips read vs. manual.
- **Clip editing** (arrange interaction layer): the arrange header carries a **grid-snap picker** (Off / Bar / Beat / 1/2 / 1/4, default Bar; **Bar follows the meter** — 3 beats in 3/4), a themed chip menu styled like the input-device chip, right-aligned under an "ARRANGE" label. Snap governs move, trim, and split (never fades — a fade is a continuous shape). A clip is **direct-manipulation**: drag the body to move, the ~6 pt left/right edge strips to trim, the top-corner grips to fade, double-click at a point to split (a context menu also offers Split at Playhead + curve toggles + Reset Gain). Fades render as **translucent** shading (never opaque — the waveform reads through) dimming the faded corner, its inner edge the fade curve (**linear = straight, equal-power = bowed**); a small triangular grip sits at each fade's inner end, faint at rest and bright on hover/selection; alt/⌥-click a grip flips its curve. A live drag shows an SF Mono beat/length/fade readout by the cursor. Per-clip **gain** is a subtle SF Mono cyan dB chip (dB readouts are cyan) shown only when non-zero or on hover/selection; drag it up/down to adjust. **Time-stretch** is the same right edge under ⌥ (option): a faint "≈" grip cue on the right edge of an audio clip brightens cyan while ⌥ is held, and ⌥-dragging it retargets the clip's timeline length (window-invariant, wired to `stretchClip`) instead of trimming — with an SF Mono "length · ratio" readout (e.g. "6.0 beats · 1.50×"); MIDI right-edge ⌥-drag stays a plain trim (stretch is audio-only). Any non-identity clip carries a persistent SF Mono ratio/pitch **badge** ("1.50×", "+3st") so stretched clips read distinct; a clip stretched **outside the 0.75–1.5× transparent band tints amber** (border + badge glow, a soft warning — never a hard block, the ratio range still spans 0.25–4×) with a tooltip naming the ratio. While the engine's offline stretch render is pending the clip body carries an **animated shimmer** (a slow tint-colored diagonal highlight sweep — the "working" cue, never a spinner); a failed render shows a **red border + glowing red dot**. All stretch geometry/threshold/formatting is headless in `DAWAppKit.ClipStretch` (the `ClipEdit` precedent). Audio clips draw a dual-tone peak **waveform** (body + brighter core, violet when AI) computed off-main and cached by file URL, windowed by `startOffsetSeconds` so a split or leading-trim shows the right slice of the source; the outline redraws only on data change (no TimelineView, no per-frame allocation). **Clip content is never blank** (beta m10-f): a MIDI clip draws a **mini note map** — one small rounded tint pill per note, pitch mapped high-at-top across the clip height (higher note nearer the top), the beat→x/pitch→y geometry headless + tested in `DAWAppKit.MIDIMapGeometry` and SHARED by the shared `ClipMIDIMap` view with the dim take-lane note strip so the two never drift (a single-pitch clip has no vertical span, so its notes land at the bottom — the take-lane idiom; violet flows in via the tint when the clip is AI); an audio clip whose peaks are still loading off-main (or whose file is unreadable) shows a **dim flat center line** so it never reads blank while the waveform arrives (minimal on purpose — the shimmer stays reserved for "stretch working"). Both previews render in **Simple and Pro** (content is not edit chrome) and sit in the same overlay slot as the waveform, under the fade/grip decorations. Every edit routes through one of the five store clip methods, so undo/coalescing come free and mid-play gain/fade changes stay seamless. **Density**: the whole arrange surface is ONE Simple/Pro panel (`TimelineLanesView.panelID` = `"arrange"`, the shared `SimpleProToggle` sits top-right in the Arrange toolbar) — **Simple** = see clips, drag the body to MOVE, and play on a **Bar-locked grid** (an edge/corner drag moves the clip too, no dead hit-zone), with the snap picker and all trim/fade/split/gain/stretch chrome + gestures suppressed; **Pro** reveals the full clip-edit interaction layer plus the snap-resolution picker (rendered beside the chip, Pro only). The Bar lock is the pure, tested `DAWAppKit.ClipSnap.effective(density:picked:)` (it never mutates the picked resolution, so flipping back to Pro restores it — the piano-roll `effectiveSnap` idiom). Automation rows and take lanes are density-unaffected (their comps keep the raw picked snap).
- **Take lanes** (arrange comping layer, M5 iii-c): a track that has take groups grows a **takes disclosure** on its header — a stacked-layers glyph, **signal-green glowing because a group exists** (outlined while open), the take-comping sibling of the automation disclosure. Opening it grows a beat-aligned takes section directly under the clip lane (above the automation row) and a matching control panel in the sidebar (both columns share `TimelineLanesView.takeLaneRowHeight`/`takeGroupHeaderHeight`, so they stay aligned — the automation-row precedent). Per group: a small header (stacked glyph + name + "N takes" + a **FLATTEN** chip, the escape hatch to ordinary clips) over **one sub-row per lane**. A lane row draws its take material **dim** (a faded peak waveform for audio, faded note pills for MIDI, windowed/positioned by the lane clip's absolute beats via the shared `ClipWaveform` / `ClipMIDIMap` idiom — the same note map the arrange clip block draws, faded to 0.5) with the comp's selected regions **glowing** over it (tinted rounded rects, the glow recipe — bloom under a bright core-stroke): **selected lane-range regions glow, unselected lane areas stay dim**. The **newest lane reads distinct** — a thin tint accent bar at its left edge and a bolder SF Mono name tag. **Comp painting**: drag horizontally on a lane row to paint that lane into the comp over the snapped beat range (the active `ClipSnap` grid), committing per-tick through `setCompSegments` (coalesced under `take.comp:<id>`, per-tick-safe like an automation drag); a **click without a drag selects the whole lane** (`selectTake`). On the main clip lane, each comp member carries a small SF Mono **"group · N" badge** (group name + lane count) and a thin **glowing splice line** at every internal comp boundary (a neon seam where the comp cuts from one take to the next). A member's context menu offers **Select Take N** per lane + **Flatten Group** (comp is edited via the lanes, so member clips reject the ordinary clip edits at the store); the sidebar lane rows add a per-lane select dot + delete, and a group Flatten. All geometry, paint-drag classification/snapping, the paint-override + normalize (kept inside what `setCompSegments` accepts), and the badge/splice derivations are headless in `DAWAppKit.TakeComp`/`TakeLaneGeometry`/`TakeLaneSelection` (the `ClipEdit`/`AutomationLane` precedent); the view is thin over them and every mutation routes through a store take method (undo/coalescing free). Violet is untouched here — takes are recorded/human unless a lane clip is itself AI-generated, in which case its material tints violet as always.
- **Sketchpad** (AI song-generation panel, M6 iii-b): the app's front door to full-song generation, and the clearest expression of the **violet = AI-generated** rule — *everything* in the panel that produces or represents generated audio carries `DAWTheme.ai`. Opened by a violet **SKETCHPAD** header chip (a `sparkles` glyph, cyan-off/violet-lit-and-glowing when open, the input-device-chip idiom), it docks as a fixed-width (340 pt) glass panel on the RIGHT of the Arrange workspace — beside the timeline on purpose, so an imported take's new violet clip lands in view. A faint violet hairline edges the whole panel (this is the AI surface). Anatomy top→bottom: a header (violet status dot + "AI SKETCHPAD" + close), an optional **sidecar banner** (only when the local ACE-Step sidecar isn't ready — amber for installed-not-running with an inline **START** button / not-installed, red for a load error, a first-class **cyan `.progress` tone** while starting — M10-b: a small `ProgressView` spinner replaces the static glyph, the message composes "Starting — ‹phase› (‹N›s)" from the sidecar's phase hint + elapsed seconds, and the Start-button slot becomes a disabled spinner + "STARTING…" control; healthy shows no banner), then the **composer** — a "STYLE" prompt editor, a "LYRICS" editor with a row of violet **section-tag buttons** (`[verse] [chorus] [bridge] [outro]`, the bracketed-structure helper that inserts a tag on its own line at the caret), a cyan SF Mono **length stepper** (−/＋ around a `30s` readout, clamped 15–240 s — cyan because it's a numeric readout, not an AI signal), and a full-width **GENERATE** button that is **violet + glowing when armed** (a healthy sidecar and a non-blank prompt) and dim/disabled otherwise. Below a hairline, the **CANDIDATES** list (newest first): each candidate is a violet-accented glass card whose body follows its lifecycle — **queued** (a dim "QUEUED" + idle bar), **generating** (an SF Mono `STEP n/m` + `NN%` with a violet glowing progress bar UNDER a slow violet **shimmer** sweep — the ClipShimmer "working" cue, never a spinner), **succeeded** (SF Mono `NN BPM · NNs LEN` readout, a cyan **PREVIEW** play/stop button — an `AVAudioPlayer` in the app layer, off the engine, one candidate at a time — and a violet glowing **IMPORT** button), **failed** (the card's accent flips to red: red waveform glyph, red border, the worker's message, a dismiss ✕), and **imported** (a violet `checkmark.seal` badge + "IMPORTED → ‹track name›"). A transient poll or import error never kills a card — it dims to 60 % and shows a small amber **RECONNECTING** tag while it keeps polling (the model's stale-tolerance rule). All state and every transition live headless in `DAWAppKit.SketchpadModel` (generate/refresh/import + the sidecar→banner mapping + the section helper), tested against a fake `SongGenerating`; the view owns only the two things the model must not — the 1 s poll cadence (a `.task` loop, active only while a candidate is still generating) and the preview player. Import routes through `ProjectStore.importGeneration`, so the created track + clip are AI-flagged (violet) and tempo-adoption is one undo, for free.
- **Clip vocal-fix panel** (AI region repaint, M6 v-b-2): a second violet AI surface, opened by a violet **FIX WITH AI** affordance that appears in the arrange header whenever a plain AUDIO clip is selected (the `wand.and.stars` glyph, dim off / violet-lit-and-glowing when the panel is open — the SKETCHPAD-chip idiom). It docks as a fixed-width (340 pt) glass panel on the RIGHT of the Arrange workspace beside the timeline (so the resulting violet take LANE lands in view), a faint violet hairline edging the whole panel. Anatomy top→bottom: a header (violet dot + "FIX WITH AI" + close), then the **composer** — the selected clip's name (with a signal-green waveform glyph), a **REGION TO FIX (BEATS)** pair of **cyan SF Mono** editable START/END fields (beats are numeric readouts, so cyan not violet — the Sketchpad length-stepper rule), a "WHAT TO FIX" prompt editor + optional "LYRICS" editor, a **STRENGTH** segmented chip row with beginner-readable labels (**SUBTLE / BALANCED / BOLD** → conservative/balanced/aggressive, the active half **violet**-lit because it's an AI parameter, Balanced default), and a full-width **FIX THIS REGION** button that is **violet + glowing when armed** (a target clip + a well-formed region) and relabels to **WORKING…** while a submit is in flight. Below a hairline, the **AI FIXES** jobs strip (newest first): each fix is a violet-accented glass card labelled by its region ("beats 33–41", SF Mono) whose body follows its lifecycle — **pending** (dim "QUEUED" + idle bar), **running** (an SF Mono `STEP n/m` + `NN%` over a violet glowing progress bar UNDER a slow violet shimmer sweep — the shared ClipShimmer "working" cue, never a spinner), **succeededAwaitingImport** (a green "READY" + a violet glowing **IMPORT** button), **imported** (a violet `checkmark.seal` badge + "→ ‹lane name›"), **failed** (accent flips **red**: red border + worker message + dismiss ✕), and **stale** (accent flips **amber**: a warning triangle + the drift message + dismiss ✕ — the target moved/re-stretched/tempo-changed since submit, so it must be re-run). A transient poll/import blip never kills a card — it dims to 60 % and shows an amber **RECONNECTING** tag while it keeps polling (the Sketchpad stale-tolerance rule; distinct from the terminal amber `stale` state). All state + transitions live headless in `DAWAppKit.ClipFixModel` (the `SketchpadModel` precedent: submit/refresh/import + the store-error → card-state mapping), tested against injected fakes; the view owns only the 1 s poll timer (a `.task` loop, active only while a job is still running) and re-seeds the composer as the audio selection changes. Submit routes through `ProjectStore.fixClipRegion`, import through `importClipFix`, so the take lands as a violet AI lane comped in over exactly the region — the M5 take-lane UI renders the result, and one undo restores the plain clip, for free.
- **Lyrics Workshop** (AI lyrics writing, M6): a violet **WRITE WITH AI** disclosure nested INSIDE the Sketchpad composer, directly above the LYRICS editor it feeds — violet is correct here, every word it produces is AI-authored, so the whole block sits on a faint violet wash with a violet hairline (the Sketchpad's own AI surface, one level in). Collapsed it's a single header row (a `sparkles` glyph + "WRITE WITH AI" + a chevron, with a small SF Mono **"via ‹provider›"** credit once a draft exists). Expanded, anatomy top→bottom: a **THEME** editor (what the song is about — required) and an optional **STYLE** editor (genre/feel), a **STRUCTURE** chip row (each section tag a violet capsule with an ✕ to remove, a **RESET** control, and a `+ verse / + chorus / + bridge / + outro` quick-add row — the section vocabulary the ACE-Step bracketed format uses), a full-width **WRITE** button that is **violet + glowing when armed** (a non-blank theme) and relabels to **REWRITE** once a draft exists, and — only after a first draft — a **refine** field + **REFINE** chip (revise the current draft by instruction) plus a "Draft ready" footer with a violet glowing **APPLY TO LYRICS** button that pushes the bracketed draft into the Sketchpad's LYRICS editor below. A failure (including "no provider configured") shows a red error strip naming the fix; while a write is in flight the button shows an inline "WRITING…" spinner and the controls dim. The workshop is **project-aware** — it weaves the live key/tempo/time-signature into the prompt so the words fit the session. All state/gating/structure-editing live headless in `DAWAppKit.LyricsWorkshopModel` (the `SketchpadModel` precedent), tested against a fake `LyricsGenerating`; the view is thin over it, and provider selection + the key chain stay in AIServices (`resolveLyricsWriter`, Anthropic-first → OpenAI, never a key value in the UI).
- **Copilot rail** (AI chat, M6 rail-d): the app's most AI-identified surface — a right-docked (360 pt) violet-edged dark-glass rail over the app-level `CopilotEngine` (rail-c), where you tell the AI what you want in plain language and it operates the project through the SAME control-command surface the WebSocket uses, one undoable step at a time. Unlike the Sketchpad/ClipFix panels (Arrange-workspace panels beside the timeline), the copilot is **app-level, not selection-gated**: its violet **COPILOT** header chip (a chat-bubble glyph, dim off / violet-lit-and-glowing when open — the SKETCHPAD-chip idiom) is **always visible**, and the rail docks **outermost-right spanning the full workspace height in BOTH Arrange and Mix**. A faint violet hairline edges the whole rail (the AI surface). Anatomy top→bottom: a header (violet glowing identity dot + "COPILOT" + a dim mono "AI" credit + a **reset** ↺ that clears the conversation, shown once it's non-empty, + close ✕), a scrolling **transcript** (auto-scrolls to the newest entry / the working cue), and an **input row** (a dark SF-Pro field + a violet glowing **send** arrow that's dim/disabled until the draft is non-blank; while a turn runs the field disables to "Working…" and the send swaps to a **red stop** cancel — work already applied survives, each step undoable). Empty, the transcript shows a beginner-readable **first-use hint** ("Ask the copilot" + plain-language example prompts in mono violet quotes — the Simple-by-default rule). Entry rendering, styled per kind so the conversation reads at a glance: **user** turns are right-aligned neutral bubbles; **assistant** prose is left, violet-tinted glass with a `sparkles` glyph (AI is the identity); a **toolCall** is a compact violet chip — the dotted command in mono violet + a one-line arg summary; a **toolResult** is a chip with a **semantic left edge — green ok / red error** (the fastest outcome read) + a check/x glyph + the result summary (red text on error); a **failure** is a red-tinted strip with the actionable, key-free message. While a turn runs, a violet **WORKING** shimmer row (the SketchpadShimmer sweep — never a spinner) sits at the transcript tail, one per round. The engine owns all conversation state (transcript/status/turn loop, its own 15-test suite); the rail is thin over it (the ClipFixPanel precedent), owning only the local draft text and a transient synchronous-`send()` error (e.g. "no AI provider configured"), shown as a red strip above the input — never a transcript entry. Chip summaries are compacted to one line by the headless, unit-tested `DAWAppKit.CopilotSummaryFormat` (the `ClipStretch` idiom). `debug.copilotSeed {mode: conversation|failed|idle}` stages the rail for captures/E2E via the engine's `seedForCapture` hook (no provider call — the `debug.clipFixSeed` precedent).
- **Explain this** (in-app plain-language help, M8 ex-a mechanism, ex-b coverage; VISION.md pillar 4): an **explain MODE**, not always-on tooltips. A violet **EXPLAIN** header chip (a `questionmark.circle` glyph, dim off / violet-lit-and-glowing when active — the SKETCHPAD/COPILOT header-chip idiom) toggles the mode (sticky until pressed again or **Esc**); it is violet because it's an AI-identified affordance (Rule 3 — violet = AI only). While active, hovering any registered control floats a **card**: a violet-edged dark-glass popover (~300 pt) anchored beside the control (below when there's room, else above; a short ~220 ms dwell in, a ~260 ms grace out so the pointer can reach it) with a violet `?` glyph + **control name** (SF Pro semibold) + **2–3 beginner-readable sentences** (WHAT it does + WHEN you'd reach for it) + a footer **"Ask the Copilot →"** button (violet). Explain is an **overlay, not a lockout** — clicks pass through to the control underneath, and normal `.help()` tooltips are unaffected. The hand-off closes explain mode, opens the **Copilot rail**, and prefills its input with *"Explain ‹control name› — what should I do with it here?"* — a **draft only** (never auto-sent, so it works key-or-no-key), routed through the rail's existing draft path (no engine change). **Copy style rules** (beginner test, Rule 6), enforced headless by `ExplainCatalogTests`: every registered id has an entry, title ≤ 24 chars, body 40–280 chars ending on `.`/`!`/`?`, and **no raw unit jargon** a newcomer wouldn't know (dB / Hz / …) — spell concepts out ("beats per minute", "louder / quieter"). **Architecture**: the curated copy is headless in `DAWAppKit.ExplainCatalog` (`[ExplainID: ExplainEntry{title, body}]`, the `PanelDensity` precedent) so previews, the app, and the tests share one source of truth; a `.explainable(_ id: ExplainID)` SwiftUI modifier (DAWApp view chrome) reports each control's frame to an app-level `ExplainCoordinator` while the mode is on, and the transient on/off + capture-focus state lives in `DAWAppKit.ExplainModel` (NOT persisted — a session aid, unlike density). **Per-instance anchoring** (ex-b): the card anchors on the HOVERED instance's own frame, so a control that renders many times (every mixer strip, every track row, every clip, every provider key row) carries ONE shared `ExplainID` and still lands its card on whichever copy the pointer is over — no per-instance frame collisions. The id-keyed frame dict is kept only as the deterministic fallback for `debug.explainMode {focus}`, which names an id the wire can't hover. `debug.explainMode {on, focus?}` stages the mode/card for captures & E2E (app-level, debug tier — off `allCommands`/MCP, the `debug.panelDensity` precedent; `focus` names an `ExplainID` to present a card without a synthesized hover). **Coverage is app-wide (ex-b)**: transport, the whole mixer console (every channel + bus strip + the master), the piano roll (density chip, snap, editor surface, velocity lane), the arrange surface (SNAP, add-track, track rows, clip body), the AI panels (Copilot/Sketchpad/Fix chips + their composer controls, the Lyrics Workshop), and Settings (the API-key rows — copy never surfaces a key value). **Shared controls carry ONE id reused wherever they render** (the SIMPLE/PRO chip → `panelDensity` across all four panels; a track row's Mute/Solo/Arm → the mixer's ids) — never two entries with near-identical copy. Where chrome is pure Canvas/gesture (the clip body's fade grips, trim edges, ⌥-stretch; the piano-roll grid), ONE entry honestly summarizes the surface's affordances rather than faking per-grip tags. **Every NEW control ships with an `ExplainCatalog` entry** — enforced headless by a completeness check plus a count floor that fails a silent shrink.
- **Onboarding tour** (the "first song in ten minutes" guided flow, M8 ob-b; VISION.md pillar 4): a seven-step coach-mark tour that walks a first-time user from an empty project to a saved song. **The chrome is PRODUCT, not AI — so it carries NO VIOLET (Rule 3), unlike the Explain/Sketchpad/Copilot surfaces.** It is a **glass step card** in the neutral palette with a **cyan active-state accent**: a progress header ("GUIDED TOUR" + 7 dots, the current dot cyan-lit + glowing, done dots dim-cyan, upcoming faint, + an "N / 7" mono readout), the step **title** (SF Pro semibold) + a **beginner body** (the same Rule-6 copy contract as Explain — plain language, no unglossed unit jargon, sentence-final punctuation, enforced headless by `OnboardingModelTests`), a **primary CTA** (a cyan-filled button — the step's natural action), a quiet **"Skip step"** (task steps only), and a quieter **"Skip tour"**. Even the `done` card — whose body *talks about* Explain and the Copilot — keeps the non-violet chrome. **Placement**: `welcome` / `done` are **centered** framing cards (no anchor); the five task cards **anchor coach-mark-style** beside their step's control, reusing the Explain frame machinery (`ExplainFramePreferenceKey` / `ExplainCoordinator`) — the `.explainable` reporters flow frames while a tour step is active (not just in explain mode). A glowing **cyan highlight ring** marks the anchored control (never covers it), the card flips **above** its anchor near a screen edge, and when the anchor isn't on screen (e.g. the `mix` step's fader while in Arrange) the card floats **center** until the CTA reveals it, then re-anchors. The overlay's empty area is **click-through**, so the user operates the very controls the tour points at — the real operation fires the completion signal that advances the step. **Offer-once + replay**: on first launch, if the tour is eligible (`shouldOfferTour`) the welcome card IS the offer; "Skip tour" dismisses it forever (the two terminals never re-offer), and it is **replayable** from a neutral **"Replay tour"** row in Settings (`reset()` → `begin()`). Onboarding progress is an app-side preference (UserDefaults key `onboarding.state`), never project data. **Signal doctrine — an app-side OBSERVING ADAPTER, not scattered emission** (`OnboardingSignalAdapter`, owned by AppModel): the tour model NEVER reads `ProjectStore`; the adapter observes four store facts and translates them to signals — tracks-with-clips 0→>0 → `projectGainedContent`, `transport.isPlaying` false→true → `playbackStarted`, a new `lastEditEvent` (classified by the pure `OnboardingEditClassifier`: level/pan/master/preset → `mixerAdjusted`, everything else → `editPerformed`, so **a fader move never reads as the `shape` step**) → `editPerformed`/`mixerAdjusted`, and a `renderCompletedCount` increment → `renderCompleted`. Observing once catches BOTH UI and control-wire actions (the ob-c requirement); the model's strict active-step matching makes multi-signal actions benign (a skeleton apply fires two signals; only the matching one advances). ## Panel splitters

The arrange workspace is **adjustable** (beta item m10-d): three dimensions resize by dragging, each persisted so a shaped layout survives relaunch. **Sidebar width** (drag the vertical splitter between the track list and the timeline, clamped 250–420 pt), **bottom-editor height** (drag the horizontal splitter on the editor's top edge, stored as a FRACTION of window height, clamped 0.30–0.55), and the **global track-row height** (drag the bottom edge of any track header, clamped 24–64 pt). Row height is ONE global value, not per-track: the sidebar headers and the timeline lanes read the SAME `PanelLayoutStore.rowHeight` in the same expression, so they stay pixel-aligned at every size (only the base clip lane scales — the automation row and take sub-rows keep their own constants and compose on top). Layout is an app-side sticky **PREFERENCE**, never project data — persisted as `panelLayout.<dimension>` in UserDefaults, never written into the project file (the density precedent). **The clamp ranges are conservative on purpose**: the reachable extremes must not visibly break at the window the app runs at. The sidebar min (250) is the track-header row's measured intrinsic floor (the fixed-width meter + M/S/A chips don't priority-shrink yet); the editor max (0.55) keeps the app header + arrange chrome + transport visible at the height the app opens at. **Real small-window adaptivity — scroll / priority-shrink when the window itself is small — is roadmap item m10-j**; until it lands a genuinely small window can still overflow at an extreme value, which is why the ranges (not the drawing) hold the line.

A splitter is a **dark-glass hairline that earns its accent by state** (Rule 3): a quiet `gridEmphasis` line at rest, cyan-lit + glowing ONLY while hovered or dragging — never violet. Its hit target is ≥ 6 pt even though it draws as a 1 px hairline, so it is easy to grab, and it holds its resize cursor for the whole drag (the m10-c gesture-driven cursor idiom) so the cursor persists when the pointer leaves the strip. A **per-row grabber is idle-invisible** (no rest hairline) so N stacked track rows don't grow N rest lines — it is discovered by the resize cursor + hover glow alone; panel-boundary splitters (sidebar, editor) show their rest hairline.

**Cursor convention — labeled decision**: a height adjustment (editor top edge, track-header bottom edge) shows **`resizeUpDown`**, NOT the open-hand grab the beta note loosely called for. Up/down resize is the macOS-correct glyph for "drag to change a height/amount" (the Pointer-affordances value-adjuster rule); grab means "pick up and move an object," which a divider is not. Width adjustment (sidebar splitter) shows `resizeLeftRight`. **Architecture**: values + clamping are headless in `DAWAppKit.PanelLayoutStore` (`@Observable`, injected `PanelLayoutBacking`, the `PanelDensityStore` precedent — clamping lives in the store so a raw drag delta is tamed); the reusable `DAWApp/Components/PanelSplitter` takes plain drag closures (axis + cumulative translation) so any resizable boundary wires to it. `debug.panelLayout {sidebarWidth?, editorFraction?, rowHeight?, reset?}` stages the layout for captures/E2E (app-level, debug tier — off `allCommands`/MCP, echoes the post-clamp values). Like pointer feel, adjustable layout is a documented **non-invokable** window preference — no wire command / MCP tool.

## Pointer affordances

Every drag/resize surface **advertises its controllability by changing the mouse cursor on hover** (beta item m10-c) — a beginner should discover what is adjustable just by moving the pointer, without a manual (Rule 6). Cursors follow ONE convention per control family; pick the family, never hand-code a per-control cursor:

- **Vertical value drags → `resizeUpDown`.** Long-throw faders (master + channel), rotary knobs (pan), velocity stems, the clip-gain chip. A value adjuster keeps its resize cursor for the WHOLE drag — a fader never "grabs" (it is not a movable object, it is a value). Convention chosen over open-hand because up/down resize reads as "drag to change the amount," the most literal cue for a beginner.
- **Horizontal value / resize drags → `resizeLeftRight`.** Clip trim edges, note right-edge resize handles, clip fade-corner grips, the send mini-fader, the ⌥ time-stretch handle. Anything dragged left/right to resize or set a horizontal amount.
- **Movable bodies → `grab` (open hand) at rest, `grabbing` (closed hand) while dragging.** Arrange clip bodies, piano-roll notes, automation breakpoints. The hand OPENS on hover and CLOSES for the whole drag, even when the pointer leaves the object's bounds (driven by the gesture, not the hover).
- **Place / paint surfaces → `crosshair`.** Empty automation-lane space (click to add a breakpoint), take-lane comp painting. Distinct from `grab` because you are authoring content by position, not picking up an existing object.
- **Text-editable fields stay the system I-beam** — never override it.

**Architecture**: the pure vocabulary and per-surface catalog are headless in `DAWAppKit.CursorKind` / `CursorAffordance` (with `forClipZone` / `forNoteZone` mirroring the geometry classifiers), so the mapping is tested once and every view reads the same contract; DAWApp resolves `CursorKind → NSCursor` in one thin switch and applies it via the `.hoverCursor(_:)` / `.hoverCursor(resolve:)` modifier (`Components/HoverCursor.swift`). The modifier is a transparent, hit-test-through `NSTrackingArea` overlay — it COEXISTS with the Explain hover machinery and the view's own gestures, and it restores the arrow on hover-exit and teardown so a cursor never sticks when a hovered view disappears or a panel opens mid-hover. During a drag the cursor is forced by the gesture's `onChanged`/`onEnded` (not hover) so it holds outside the bounds and returns to the correct rest state on release. Pointer feel is a documented **non-invokable** convention — no wire command / MCP tool (the sp-b pure-view precedent).

## Rules

1. Dark theme is the product; light theme is out of scope until M9+.
2. No stock gray AppKit controls in the main window — custom or restyled.
3. One accent color per meaning; never reuse cyan for anything but playback/activity. Create/add affordances (the "+" glyphs) are **neutral chrome** — `textPrimary` on the raised chip, no accent and no glow at rest (the standard hover-brighten is fine); an accent is earned by state, not by inviting a click.
4. Animation: 120–200ms ease-out for state; meters/positions animate continuously, chrome never bounces.
5. Text contrast ≥ 4.5:1 for labels; readouts may glow but must pass at core color.
6. Beginner test: every panel understandable by a newcomer in Simple mode without a manual. If a label needs a manual, rewrite it.
