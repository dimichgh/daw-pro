# Design Language — "Glass Cockpit"

Dark, glassy, information-dense but calm. Every glow means something. Think: night-flight avionics meets high-end studio hardware, rendered with macOS-native polish.

## Foundations

- **Surfaces**: near-black layered glass. Base `#0B0D12`, raised panels `#12151D` with 1px `white @ 6%` hairline borders and subtle inner shadow. Use translucency (`.ultraThinMaterial`-style) sparingly, only for overlays.
- **Text**: SF Pro for labels; **SF Mono for every numeric readout** (time, tempo, dB, Hz). Readouts are "digital": mono font, slight letterspacing, colored glow. Text runs a measured hierarchy — `textPrimary` (primary content) › `textSecondary` (legible secondary labels, ≥ 4.5:1) › `textDim` (muted captions/micro-labels, ≥ 4.5:1) › `textFaint` (placeholder & decorative floor, ≥ 3.0:1); every value is measured against the raised-panel surface. Never dim a real label below 4.5:1 with ad-hoc `.opacity()` — reach for the right token.
- **Accent system** (semantic, not decorative):
  - Cyan `#3EE6FF` — playback/position/active state
  - Amber `#FFB84D` — record/warning/clipping-adjacent
  - Red `#FF4D5E` — clip/overload/destructive
  - Green `#5DFF9F` — healthy signal, success
  - Violet `#B48CFF` — AI/generated content (everything AI-touched carries violet)
- **Glow recipe**: color at full opacity core + same-color shadow (radius 6–10, opacity 0.5–0.7) + faint wide bloom (radius 20, opacity 0.15). Glow intensity may encode magnitude (meter brightness follows level). Never glow static chrome.

## Components

- **Meters**: segmented LED-style bars (Canvas-drawn), green→amber→red zones, peak-hold tick that decays, glow tracks level. 60fps via `TimelineView`.
- **Transport**: oversized readout cluster — bars.beats + SMPTE time in mono cyan, tempo editable inline, glowing play/record buttons with state halos.
- **Knobs/faders**: flat dark caps, thin neon value arcs/tracks; value readout appears on touch. Double-click resets. Fine-drag with ⌥.
- **Waveforms**: dual-tone (body + brighter RMS core), violet tint when clip is AI-generated.
- **Panels**: every panel has Simple and Pro density modes; Simple shows the 20% of controls used 80% of the time with plain-language labels ("Punchier" over "Ratio 4:1" — Pro mode shows both). Mode is a themed segmented chip pair (SIMPLE / PRO), the active half cyan-lit; Simple is the default. **Density is a per-panel, app-sticky PREFERENCE, never project data** — each panel owns its own mode (a beginner may want a Pro piano roll but a Simple mixer), it survives close/reopen and app relaunch (persisted as `panelDensity.<panelID>` in UserDefaults), and it is NEVER written into the project file. The chip pair is one shared component — `DAWApp/Components/SimpleProToggle`, bound to a `DAWAppKit.PanelDensityStore` + a stable panel ID — so it is pixel-identical everywhere (SF Mono 9 pt bold uppercase, tracking 0.6, active half `playback` on an 0.18 fill, on a raised chip with a hairline border). The headless model lives in `DAWAppKit.PanelDensity`/`PanelDensityStore` (injected persistence backing, `@Observable`); `debug.panelDensity {panel, mode}` stages a mode for captures/E2E (debug tier — not an MCP tool). The per-panel Simple/Pro plan lives in `docs/research/simple-pro-inventory.md`. Panels that already pass the beginner test adopt the model but render identically in both modes (their modes coincide); the piano roll is the reference implementation. As of sp-d, **density coverage is uniform**: five surfaces carry live chips (piano roll, mixer, arrange, transport, and — m10-n-3 — the **instrument picker**, whose Simple "Instrument Sets" and Pro full browser genuinely differ); every other panel's Simple and Pro modes **COINCIDE by design and must never grow a do-nothing toggle** — a toggle that changes nothing is worse than none (the per-surface roll-up lives in `docs/research/simple-pro-inventory.md` §"Density adoption status"). A new surface earns a chip ONLY when its two modes genuinely differ (the picker does; the inventory records the verdict).
- **Arrange timeline**: fixed pixels-per-beat (16 pt/beat, no zoom yet); one lane per track in sidebar order. Clips are rounded tinted blocks — signal-green for audio, playback-cyan for MIDI, **violet whenever AI-touched** — with a name label when width allows, selected clip brighter + glowing. Beat grid is hairlines; bar lines brighter; bar numbers in SF Mono. Playhead is a glowing cyan hairline driven by transport position (offset a view, never a per-frame Canvas redraw). The **loop region** (beta m10-g) is a direct-manipulation band in a distinct strip at the TOP of the ruler (`y ∈ [2, 14]` of the 56 pt ruler — the ruler grew 42 → 56 in m11-c to seat the marker lane below the loop band, still clear of the bar numbers and read-through of the playhead): **cyan glowing when looping is enabled** (cyan = earned active transport state, NEVER violet), a **dim low-opacity outline when a region exists but looping is off**, nothing when there is no region. The whole ruler height is the interaction surface (so the loop is grabbable anywhere in the ruler column): **drag empty ruler to sketch** a new region (snapped to the arrange grid — Bar in Simple, the picked snap in Pro — with a minimum of one snap unit, a sub-threshold drag falling through to a click), **drag either edge to resize** (`resizeLeftRight`, clamped one snap unit before the opposite edge — never inverts), **drag the body to move** it preserving length (grab→grabbing, clamped at beat 0), **click inside to toggle** looping (the TransportBar LOOP chip's twin), and **click empty ruler to seek** the transport to that snapped beat (the ruler is never a dead click surface — the piano-roll scrub idiom, one surface over). All zone-classification / create-resize-move / click math is headless + tested in `DAWAppKit.LoopRulerModel` (`LoopRulerGeometry` + `LoopEdit`, the `ClipEdit`/`PianoRollPlayhead` precedent); loop state flows into the timeline by value so an agent's `transport.setLoop` over the wire updates the band live, and the gestures route through the pre-existing `setLoop`/`seek` store methods — a **non-invokable pure-view surface**, no new wire command / MCP tool (the m10-c/e precedent). The **session-marker lane** (m11-c) is a SECOND direct-manipulation strip in the ruler, `y ∈ [15, 33]` — BELOW the loop band, ABOVE the bar numbers — carrying named song-section flags ("Intro", "Chorus", "Drop"). Markers are **navigation furniture, not an active or AI state, so the flags are deliberately NEUTRAL** — a dark raised-glass chip (`panelRaised` + hairline) with the name in SF Pro (`textSecondary` — a name is prose, not a numeric readout) and a thin `textSecondary` anchor bar at the marker's exact beat: **no cyan (that's earned active transport), no violet (that's AI only)**. The marker strip is the marker gesture surface, layered over the loop ruler but occupying ONLY its band, so it carves that strip out of the loop hit zone WITHOUT touching the loop band or the seek area — the m10-g loop gestures (create/resize/move/toggle/click-seek) keep working, regression-pinned. Interactions: **drag empty marker-strip → context menu "Add Marker Here"** at the snapped beat (the strip also click-seeks, so it's never dead), **drag a flag to move it** (snapped, `grabbing` cursor, the move coalesces into one undo step), **click a flag to jump** the playhead to that section (the seek path), **double-click or the context menu to rename in place** (the same SF-Pro dark-glass field + cyan focus border + trim/empty-cancel/unchanged-no-op commit rule as the track-header rename, reusing `DAWAppKit.TrackRename`), and the context menu also **deletes**. All geometry / hit-test / snapped-move / add-beat math is headless + tested in `DAWAppKit.MarkerLaneModel` (`MarkerLaneGeometry` + `MarkerLaneEdit`, the `LoopRulerModel` precedent). Both densities render markers identically (COINCIDENT-EXEMPT — see `docs/research/simple-pro-inventory.md`); markers flow into the timeline by value so an agent's `marker.*` over the wire updates the lane live, and every mutation routes through the store's marker methods (undo/coalescing free). Unlike the loop ruler this DOES ship a wire surface — five `marker.*` commands + the `transport.seek {marker}` interop (the house rule: the drag-to-move UI is a user-facing capability, so it earns a control command + MCP tool + test). **Track header identity** (beta m10-i): the sidebar row's icon+name is ALWAYS readable, and — critically — its width budget **never inflates the sidebar past its 250 pt floor** (that would slide the whole panel off the window's left edge). The name is a **SOFT, priority-won, truncating label** — `.layoutPriority(1)` on the identity cluster hands it the row's surplus so it reads (≳ 60 pt even at sidebar 250), but it carries **no hard minimum width** (its intrinsic min is just an ellipsis), so it can never push the sidebar minimum up. The `MiniLevelBar` gracefully **compresses to a narrower floor** (22–44 pt, a slimmer 10-segment bar, NEVER hidden), and the **clip-count badge is the least load-bearing chip — it folds into the identity tooltip below a sidebar-width threshold** (`DAWAppKit.TrackHeaderLayout.showsClipBadge`, folded < 300 pt) rather than steal the name's room. Row spacing tightens to 5 pt so a fully-loaded row still fits the narrow end. A long name truncates at the **tail** (the app's `lineLimit(1)` convention) with the full name + clip count in a `.help` tooltip (`TrackHeaderLayout.identityTooltip`). The heaviest row variant — a **take-group track**, which carries a SECOND disclosure — folds its **automation disclosure into the row's context menu** at a narrow sidebar (< 300 pt, `TrackHeaderLayout.showsInlineAutomationDisclosure`, m10-j), the same "drop the least load-bearing chip first" idiom as the clip badge, so the two disclosures never squeeze the soft name below its ≈56 pt readable floor (the deferred m10-i "~36 pt name at 250" edge); the takes disclosure — the control that makes the row heavy — stays visible, and automation is never lost, just relocated. A one-disclosure row always keeps automation inline. The soft name additionally guarantees nothing clips the window even at the extremes (the name just truncates further). **Double-click the identity** (or the row's **"Rename Track"** context item — the discoverability twin) inline-renames it: an SF-Pro dark-glass field (the Copilot-input idiom, a cyan-tinted focus border — a name is prose, not a numeric readout, so NOT SF Mono), **Return / focus-loss commits** via `store.renameTrack` (one journaled undo step), **Escape cancels**, and **empty or unchanged input = no-op** (no stray undo step) — the trim / empty-cancel / unchanged-no-op commit rule is headless + tested in `DAWAppKit.TrackRename` (the `CopilotSummaryFormat` precedent); while editing, the row's other click/drag side effects are inert. Adding a track is a **discoverable, labeled affordance for humans (obviously not AI-only)**: a neutral `+ ADD` chip **menu** (the themed snap-picker idiom, `textPrimary` create-chrome per Rule 3) offering **Instrument / Audio / Bus**, with a full **`ADD TRACK`** button in the empty state — both route to `store.addTrack(kind:)` (sensible default names) and share the `.arrangeAddTrack` Explain entry; no new wire / MCP surface (`track.add`/`track.rename` pre-exist). A track's header carries an automation disclosure (axis-chart glyph, cyan-lit + glowing when the track has an enabled non-empty lane); opening it grows a beat-aligned breakpoint-editor row under the clip lane and a matching control panel in the sidebar (so both columns stay aligned). The editor is a Canvas neon polyline (glow recipe: wide faint bloom under a bright core) with glowing breakpoint dots and a faint dashed neutral guide (unity gain / center pan); **volume lanes track the fader's cyan, pan lanes stay neutral white** (mixer semantics — pan claims no accent), a **disabled lane renders dimmed** (its curve is inert; the fader/knob is manual). Click empty space adds a point, drag moves it with a live SF Mono value/beat readout by the cursor (dB for volume, L/C/R for pan), double-click or select+⌫ deletes; edits redraw only on interaction (no TimelineView). Target picker is volume/pan in v0 (a themed VOL/PAN chip pair, the active half cyan-lit, a dot marking which already has a lane); an enable chip (green ON / dim OFF) flips read vs. manual.
- **Clip editing** (arrange interaction layer): the arrange header carries a **grid-snap picker** (Off / Bar / Beat / 1/2 / 1/4, default Bar; **Bar follows the meter** — 3 beats in 3/4), a themed chip menu styled like the input-device chip, right-aligned under an "ARRANGE" label. Snap governs move, trim, and split (never fades — a fade is a continuous shape). A clip is **direct-manipulation**: drag the body to move, the ~6 pt left/right edge strips to trim, the top-corner grips to fade, double-click at a point to split (a context menu also offers Split at Playhead + curve toggles + Reset Gain). Fades render as **translucent** shading (never opaque — the waveform reads through) dimming the faded corner, its inner edge the fade curve (**linear = straight, equal-power = bowed**); a small triangular grip sits at each fade's inner end, faint at rest and bright on hover/selection; alt/⌥-click a grip flips its curve. A live drag shows an SF Mono beat/length/fade readout by the cursor. Per-clip **gain** is a subtle SF Mono cyan dB chip (dB readouts are cyan) shown only when non-zero or on hover/selection; drag it up/down to adjust. In **Pro**, a SELECTED audio clip additionally reveals a **gain-envelope overlay** (m13-e) — a cyan Canvas polyline of glowing breakpoint dots (the glow recipe) that rides the clip's volume up and down over time, over a faint dashed 0 dB guide; click to add a point, drag to move (live SF Mono dB/beat readout), double-click to delete. The overlay draws **no opaque backing** — just the polyline + dots + guide — and **the waveform stays visible UNDER it as a 0.4-opacity GHOST** (m13-g sub-task 3) so breakpoints read against the audio they shape (cyan overlay is the primary layer, ghosted waveform the backdrop). One `.explainable(.clipGainEnvelope)` card summarizes the whole Canvas/gesture surface (the honest-scope rule). **Simple hides the overlay entirely** — the waveform then draws at full strength. **Time-stretch** is the same right edge under ⌥ (option): a faint "≈" grip cue on the right edge of an audio clip brightens cyan while ⌥ is held, and ⌥-dragging it retargets the clip's timeline length (window-invariant, wired to `stretchClip`) instead of trimming — with an SF Mono "length · ratio" readout (e.g. "6.0 beats · 1.50×"); MIDI right-edge ⌥-drag stays a plain trim (stretch is audio-only). Any non-identity clip carries a persistent SF Mono ratio/pitch **badge** ("1.50×", "+3st") so stretched clips read distinct; a clip stretched **outside the 0.75–1.5× transparent band tints amber** (border + badge glow, a soft warning — never a hard block, the ratio range still spans 0.25–4×) with a tooltip naming the ratio. While the engine's offline stretch render is pending the clip body carries an **animated shimmer** (a slow tint-colored diagonal highlight sweep — the "working" cue, never a spinner); a failed render shows a **red border + glowing red dot**. All stretch geometry/threshold/formatting is headless in `DAWAppKit.ClipStretch` (the `ClipEdit` precedent). Audio clips draw a dual-tone peak **waveform** (body + brighter core, violet when AI) computed off-main and cached by file URL, windowed by `startOffsetSeconds` so a split or leading-trim shows the right slice of the source; the outline redraws only on data change (no TimelineView, no per-frame allocation). **Clip content is never blank** (beta m10-f): a MIDI clip draws a **mini note map** — one small rounded tint pill per note, pitch mapped high-at-top across the clip height (higher note nearer the top), the beat→x/pitch→y geometry headless + tested in `DAWAppKit.MIDIMapGeometry` and SHARED by the shared `ClipMIDIMap` view with the dim take-lane note strip so the two never drift (a single-pitch clip has no vertical span, so its notes land at the bottom — the take-lane idiom; violet flows in via the tint when the clip is AI); an audio clip whose peaks are still loading off-main (or whose file is unreadable) shows a **dim flat center line** so it never reads blank while the waveform arrives (minimal on purpose — the shimmer stays reserved for "stretch working"). Both previews render in **Simple and Pro** (content is not edit chrome) and sit in the same overlay slot as the waveform, under the fade/grip decorations. Every edit routes through one of the five store clip methods, so undo/coalescing come free and mid-play gain/fade changes stay seamless. **Density**: the whole arrange surface is ONE Simple/Pro panel (`TimelineLanesView.panelID` = `"arrange"`, the shared `SimpleProToggle` sits top-right in the Arrange toolbar) — **Simple** = see clips, drag the body to MOVE, and play on a **Bar-locked grid** (an edge/corner drag moves the clip too, no dead hit-zone), with the snap picker and all trim/fade/split/gain/stretch chrome + gestures suppressed; **Pro** reveals the full clip-edit interaction layer plus the snap-resolution picker (rendered beside the chip, Pro only). The Bar lock is the pure, tested `DAWAppKit.ClipSnap.effective(density:picked:)` (it never mutates the picked resolution, so flipping back to Pro restores it — the piano-roll `effectiveSnap` idiom). Automation rows and take lanes are density-unaffected (their comps keep the raw picked snap).
- **Take lanes** (arrange comping layer, M5 iii-c): a track that has take groups grows a **takes disclosure** on its header — a stacked-layers glyph, **signal-green glowing because a group exists** (outlined while open), the take-comping sibling of the automation disclosure. Opening it grows a beat-aligned takes section directly under the clip lane (above the automation row) and a matching control panel in the sidebar (both columns share `TimelineLanesView.takeLaneRowHeight`/`takeGroupHeaderHeight`, so they stay aligned — the automation-row precedent). Per group: a small header (stacked glyph + name + "N takes" + a **FLATTEN** chip, the escape hatch to ordinary clips) over **one sub-row per lane**. A lane row draws its take material **dim** (a faded peak waveform for audio, faded note pills for MIDI, windowed/positioned by the lane clip's absolute beats via the shared `ClipWaveform` / `ClipMIDIMap` idiom — the same note map the arrange clip block draws, faded to 0.5) with the comp's selected regions **glowing** over it (tinted rounded rects, the glow recipe — bloom under a bright core-stroke): **selected lane-range regions glow, unselected lane areas stay dim**. The **newest lane reads distinct** — a thin tint accent bar at its left edge and a bolder SF Mono name tag. **Comp painting**: drag horizontally on a lane row to paint that lane into the comp over the snapped beat range (the active `ClipSnap` grid), committing per-tick through `setCompSegments` (coalesced under `take.comp:<id>`, per-tick-safe like an automation drag); a **click without a drag selects the whole lane** (`selectTake`). On the main clip lane, each comp member carries a small SF Mono **"group · N" badge** (group name + lane count) and a thin **glowing splice line** at every internal comp boundary (a neon seam where the comp cuts from one take to the next). A member's context menu offers **Select Take N** per lane + **Flatten Group** (comp is edited via the lanes, so member clips reject the ordinary clip edits at the store); the sidebar lane rows add a per-lane select dot + delete, and a group Flatten. All geometry, paint-drag classification/snapping, the paint-override + normalize (kept inside what `setCompSegments` accepts), and the badge/splice derivations are headless in `DAWAppKit.TakeComp`/`TakeLaneGeometry`/`TakeLaneSelection` (the `ClipEdit`/`AutomationLane` precedent); the view is thin over them and every mutation routes through a store take method (undo/coalescing free). Violet is untouched here — takes are recorded/human unless a lane clip is itself AI-generated, in which case its material tints violet as always.
- **Sketchpad** (AI song-generation panel, M6 iii-b): the app's front door to full-song generation, and the clearest expression of the **violet = AI-generated** rule — *everything* in the panel that produces or represents generated audio carries `DAWTheme.ai`. Opened by a violet **SKETCHPAD** header chip (a `sparkles` glyph, cyan-off/violet-lit-and-glowing when open, the input-device-chip idiom), it docks as a fixed-width (340 pt) glass panel on the RIGHT of the Arrange workspace — beside the timeline on purpose, so an imported take's new violet clip lands in view. A faint violet hairline edges the whole panel (this is the AI surface). Anatomy top→bottom: a header (violet status dot + "AI SKETCHPAD" + close), an optional **sidecar banner** (only when the local ACE-Step sidecar isn't ready — amber for installed-not-running with an inline **START** button / not-installed, red for a load error, a first-class **cyan `.progress` tone** while starting — M10-b: a small `ProgressView` spinner replaces the static glyph, the message composes "Starting — ‹phase› (‹N›s)" from the sidecar's phase hint + elapsed seconds, and the Start-button slot becomes a disabled spinner + "STARTING…" control; healthy shows no banner), then the **composer** — a "STYLE" prompt editor, a "LYRICS" editor with a row of violet **section-tag buttons** (`[verse] [chorus] [bridge] [outro]`, the bracketed-structure helper that inserts a tag on its own line at the caret), a cyan SF Mono **length stepper** (−/＋ around a `30s` readout, clamped 15–240 s — cyan because it's a numeric readout, not an AI signal), and a full-width **GENERATE** button that is **violet + glowing when armed** (a healthy sidecar and a non-blank prompt) and dim/disabled otherwise. Below a hairline, the **CANDIDATES** list (newest first): each candidate is a violet-accented glass card whose body follows its lifecycle — **queued** (a dim "QUEUED" + idle bar), **generating** (an SF Mono `STEP n/m` + `NN%` with a violet glowing progress bar UNDER a slow violet **shimmer** sweep — the ClipShimmer "working" cue, never a spinner), **succeeded** (SF Mono `NN BPM · NNs LEN` readout, a cyan **PREVIEW** play/stop button — an `AVAudioPlayer` in the app layer, off the engine, one candidate at a time — and a violet glowing **IMPORT** button), **failed** (the card's accent flips to red: red waveform glyph, red border, the worker's message, a dismiss ✕), and **imported** (a violet `checkmark.seal` badge + "IMPORTED → ‹track name›"). A transient poll or import error never kills a card — it dims to 60 % and shows a small amber **RECONNECTING** tag while it keeps polling (the model's stale-tolerance rule). All state and every transition live headless in `DAWAppKit.SketchpadModel` (generate/refresh/import + the sidecar→banner mapping + the section helper), tested against a fake `SongGenerating`; the view owns only the two things the model must not — the 1 s poll cadence (a `.task` loop, active only while a candidate is still generating) and the preview player. Import routes through `ProjectStore.importGeneration`, so the created track + clip are AI-flagged (violet) and tempo-adoption is one undo, for free.

The window itself has a **measured minimum size** (beta item m10-j): the smallest window at which ALL fixed chrome stays visible and usable — the app header row, the arrange toolbar, a usable timeline sliver (the ruler + a row or two), the bottom editor at its max fraction when open, and the transport bar. The numbers are **measured, not guessed** (the m10-d law): each chrome row is rendered via `NSHostingView.fittingSize` at the app's real fonts/paddings and the floor derived from those intrinsics — **width 1208 pt** (bound by the Pro transport bar's measured 1183 pt inside the window's 24 pt outer padding; the transport is the widest fixed chrome and is present in both workspaces) and **height 640 pt** (derived so the editor open at its max 0.55 fraction still leaves the fixed chrome ≈189 pt + a ≳96 pt timeline sliver visible). The floor + clamp math is headless + tested in `DAWAppKit.WindowFloor`; the window enforces it via the SwiftUI root `.frame(minWidth:minHeight:)` AND the NSWindow `contentMinSize` (belt-and-suspenders), opens at a comfortable `.defaultSize` (1440×900), and `debug.windowFrame {width?, height?}` stages/echoes the window content size for captures/E2E (app-level, debug tier — off `allCommands`/MCP, the `debug.panelLayout` precedent; no-param call echoes the current frame, a size request clamps to the floor keeping the top-left fixed).

The arrange **track area scrolls as ONE unit** rather than pushing chrome off-window when content exceeds the viewport: the sidebar track rows and the timeline lanes live inside a **single shared vertical `ScrollView`** owning both columns (one HStack), so there is exactly one scroll position and the two columns can **never desync** — the structurally safer choice over synchronizing two offsets. Rows stay pixel-locked to lanes (both read `PanelLayoutStore.rowHeight`). The timeline's own **horizontal** scroll (ruler + lanes) nests inside. **The ruler block — the bar ruler, loop band, marker lane, tempo lane, and the TRACKS header — PINS ABOVE the shared vertical scroll (m13-g, audit F7)**, staying visible however deep you scroll, while the rows + lanes scroll under it. Pinning it as a **sibling above the one shared `ScrollView`** (never a second scroll) preserves the no-desync invariant — there is still exactly ONE vertical scroll owning both columns. The pinned ruler stays HORIZONTALLY synced with the lanes: the scrolling `.lanes` timeline reports its horizontal offset up through a `PreferenceKey` (`ArrangeHOffsetKey`) and the pinned `.ruler` timeline mirrors it (`offset(x: -h)`), so bar lines stay aligned across the seam. One `TimelineLanesView` renders both — an `ArrangeContent` mode (`.ruler` / `.lanes`) selects the ruler-only vs lanes-only body, so they share geometry (contentWidth / beat scale) by construction. The pinned block is fixed to the ruler height (`.frame(height: rulerHeight)`) — load-bearing: a spacer with no height constraint is vertically greedy, which would split the VStack ~50/50 and float the ruler mid-panel. Row 0 and lane 0 both anchor at the scroll's top (the ruler is pinned separately, so the lanes carry NO ruler inset). `debug.arrangeScroll {trackId?, index?, bottom?, reset?}` jumps the shared scroll to a target row (a nonce, not the id, so a repeat still fires) so a capture can frame a deep session with the block pinned. A `minHeight = viewport` on the shared content keeps the two glass panels full-height when the content is short. The Sketchpad/ClipFix side panels and the bottom piano-roll editor stay OUTSIDE the shared scroll at full height (the m10-e scrub strip is unaffected). On macOS, wheel-driven scrolling doesn't conflict with the click-drag splitters inside the scroll (the row-height grabber already lives inside a scroll). With the floor + shared scroll in place, the m10-d clamp ranges hold at every reachable value — a genuinely small window now scrolls instead of overflowing.

A splitter is a **dark-glass hairline that earns its accent by state** (Rule 3): a quiet `gridEmphasis` line at rest, cyan-lit + glowing ONLY while hovered or dragging — never violet. Its hit target is ≥ 6 pt even though it draws as a 1 px hairline, so it is easy to grab, and it holds its resize cursor for the whole drag (the m10-c gesture-driven cursor idiom) so the cursor persists when the pointer leaves the strip. A **per-row grabber is idle-invisible** (no rest hairline) so N stacked track rows don't grow N rest lines — it is discovered by the resize cursor + hover glow alone; panel-boundary splitters (sidebar, editor) show their rest hairline.

**Cursor convention — labeled decision**: a height adjustment (editor top edge, track-header bottom edge) shows **`resizeUpDown`**, NOT the open-hand grab the beta note loosely called for. Up/down resize is the macOS-correct glyph for "drag to change a height/amount" (the Pointer-affordances value-adjuster rule); grab means "pick up and move an object," which a divider is not. Width adjustment (sidebar splitter) shows `resizeLeftRight`. **Architecture**: values + clamping are headless in `DAWAppKit.PanelLayoutStore` (`@Observable`, injected `PanelLayoutBacking`, the `PanelDensityStore` precedent — clamping lives in the store so a raw drag delta is tamed); the reusable `DAWApp/Components/PanelSplitter` takes plain drag closures (axis + cumulative translation) so any resizable boundary wires to it. `debug.panelLayout {sidebarWidth?, editorFraction?, rowHeight?, reset?}` stages the layout for captures/E2E (app-level, debug tier — off `allCommands`/MCP, echoes the post-clamp values). Like pointer feel, adjustable layout is a documented **non-invokable** window preference — no wire command / MCP tool.

## Pointer affordances

Every drag/resize surface **advertises its controllability by changing the mouse cursor on hover** (beta item m10-c) — a beginner should discover what is adjustable just by moving the pointer, without a manual (Rule 6). Cursors follow ONE convention per control family; pick the family, never hand-code a per-control cursor:

- **Vertical value drags → `resizeUpDown`.** Long-throw faders (master + channel), rotary knobs (pan), velocity stems, the clip-gain chip. A value adjuster keeps its resize cursor for the WHOLE drag — a fader never "grabs" (it is not a movable object, it is a value). Convention chosen over open-hand because up/down resize reads as "drag to change the amount," the most literal cue for a beginner.
- **Horizontal value / resize drags → `resizeLeftRight`.** Clip trim edges, note right-edge resize handles, clip fade-corner grips, the send mini-fader, the ⌥ time-stretch handle. Anything dragged left/right to resize or set a horizontal amount.
- **Movable bodies → `grab` (open hand) at rest, `grabbing` (closed hand) while dragging.** Arrange clip bodies, piano-roll notes, automation breakpoints. The hand OPENS on hover and CLOSES for the whole drag, even when the pointer leaves the object's bounds (driven by the gesture, not the hover).
- **Place / paint surfaces → `crosshair`.** Empty automation-lane space (click to add a breakpoint), take-lane comp painting. Distinct from `grab` because you are authoring content by position, not picking up an existing object.
- **Text-editable fields stay the system I-beam** — never override it.

**Architecture**: the pure vocabulary and per-surface catalog are headless in `DAWAppKit.CursorKind` / `CursorAffordance` (with `forClipZone` / `forNoteZone` mirroring the geometry classifiers), so the mapping is tested once and every view reads the same contract; DAWApp resolves `CursorKind → NSCursor` in one thin switch and applies it via the `.hoverCursor(_:)` / `.hoverCursor(resolve:)` modifier (`Components/HoverCursor.swift`). The modifier is a transparent, hit-test-through `NSTrackingArea` overlay — it COEXISTS with the Explain hover machinery and the view's own gestures, and it restores the arrow on hover-exit and teardown so a cursor never sticks when a hovered view disappears or a panel opens mid-hover. During a drag the cursor is forced by the gesture's `onChanged`/`onEnded` (not hover) so it holds outside the bounds and returns to the correct rest state on release. Pointer feel is a documented **non-invokable** convention — no wire command / MCP tool (the sp-b pure-view precedent).

## Rules

1. Dark theme is the product; light theme is out of scope until M9+.
2. No stock gray AppKit controls in the main window — custom or restyled.
3. One accent color per meaning; never reuse cyan for anything but playback/activity. Create/add affordances (the "+" glyphs) are **neutral chrome** — `textPrimary` on the raised chip, no accent and no glow at rest (the standard hover-brighten is fine); an accent is earned by state, not by inviting a click.
4. Animation: 120–200ms ease-out for state; meters/positions animate continuously, chrome never bounces.
5. Text contrast ≥ 4.5:1 for labels; readouts may glow but must pass at core color.
6. Beginner test: every panel understandable by a newcomer in Simple mode without a manual. If a label needs a manual, rewrite it.
