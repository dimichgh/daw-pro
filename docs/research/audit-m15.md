## 2. Bugs and measurements, written to file

### B1 (F1) — HIGH: metronome accents follow the BASE meter, not the meter map — measured live

**Setup** (probe2.mjs, staging :17688): `project.new` → `tempo.setMap {segments:[{startBeat:0,bpm:120}], meterChanges:[{startBeat:0,beatsPerBar:4},{startBeat:8,beatsPerBar:3}]}` (accepted, mapRevision 1, readback exact) → metronome on → `debug.masterCapture` start → play from 0 → 11.5 s → stop. 552 000 frames captured.

**Analysis** (analyze-metro.mjs; deinterleaved ch0, click onsets at exact 0.5 s spacing, classified by peak amplitude AND dominant frequency — downbeat buffer is 1600 Hz @ 0.5, regular 1000 Hz @ 0.35, `Metronome.swift:19-22`):

- Expected downbeats under the map: **0, 4, 8, 11, 14, 17, 20** (bars of 3 from beat 8).
- **Measured downbeats: 0, 4, 8, 12, 16, 20** — the constant-4/4 pattern. Beats 11/14/17 got regular clicks (peak 0.333, ~1 kHz); beats 12/16 got downbeat clicks (peak 0.483, ~1.6 kHz). 23/23 clicks classified, zero ambiguity.

**Mechanism**: `AudioEngine.clickMeterMap` (`AudioEngine.swift:1607-1609`) constructs `MeterMap(constant: TimeSignature(beatsPerBar: beatsPerBar, beatUnit: 4))` from the scalar cached at `:1478` (`transport.timeSignature.beatsPerBar` — the BASE meter), and every live scheduling call passes it (`:1191/:1201/:1638/:1648`). `OfflineRenderer.swift:246` builds the same constant map. Count-in inherits the scalar separately (`AudioEngine.swift:1624` → `Metronome.swift:238-243` — a count-in anchored in a 3/4 region would click 4-beat bars; code-proven, not captured). The `Metronome` API itself is correct — m14-c proved multi-meter accents at the unit level (4/4→3/4 mid-loop test), which is precisely why this slipped: no production call site feeds it the real map.

**Fix shape** (m15-a): cache the whole `MeterMap` in `cacheTransportFlags`, delete `clickMeterMap`, pass the real map at all four live sites + OfflineRenderer + derive count-in bar length from `meterMap.beatsPerBar(atBeat: recordBeat)`. Control-plane only — zero render-thread surface.

### B2 (F2) — MED-HIGH: loop honesty while recording + the iii-b build

Probe3, live: `transport.setLoop {enabled:true, 0, 4}` → `transport.record` (armed instrument track) → recording started at the CURRENT playhead 17.096 and rolled linear (pos 25.936 four seconds later) with `isLoopEnabled: true` still reported. Three distinct issues, one owner: (1) no honesty signal that loop is suppressed while recording — an agent reading the snapshot sees a contradiction; (2) record does not seek to the loop (or punch-in) start — a user who framed a 4-beat loop to record over starts wherever the playhead last sat; (3) the actual feature — loop-cycle take capture into lanes — is now unblocked (M14's never-moving anchor; design-m13f §6.1 names the exact two suppression sites to lift together: `AudioEngine.swift:1302` + `:1720`, with `:1533` the schedule guard).

### B3 (F7) — measured: bypass toggle transition

Probe5 (gain ×4 on a sustained A3 pad, toggle mid-play, master capture): whole-file 10 ms RMS shows the swap at one window boundary (0.202→0.067); zoomed 1 ms windows put the full ×4→×1 transition inside ≤2 ms with no ramp — the documented hard swap (`EffectChainProcessor.swift:11-13`). Not new information, but now quantified: the fix (short equal-power crossfade inside the walk, atomics only) has a concrete gate — the same probe shape must show a ≥5 ms monotone ramp instead of a step.
