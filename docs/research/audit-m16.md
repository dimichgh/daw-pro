### B1 (F1) — CRITICAL: the Canvas-closure crash class, three live crashes

**Recipes** (port 17663, `.build/debug/DAWApp`, window visible):
1. `project.new` → 40 × (`track.add` + `clip.addAudio` + `track.setVolume`) at full wire speed → 16 × `fx.add` limiter → `transport.play` → **SIGSEGV** ~9 s later in `VibeMeterView.body` Canvas closure (`DAWApp-2026-07-13-070338.ips`). Same recipe re-run on a fresh instance survived (intermittent).

3. A **2-track** project (instrument + audio with fades/envelope on a deleted file) → `transport.seek` → `transport.play` → same grid-closure stack (`…-071806.ips`).

**Shared signature** (all three): main thread, `AG::Graph::UpdateStack.update` → `CanvasDisplayList.updateValue()` → `closure #1 in Attribute.syncMainIfReferences` → `swift_task_isCurrentExecutorWithFlags` → `swift_getObjectType` on a non-pointer (`0x000003e1aa010408` in crash 1). `syncMainIfReferences` walks the references captured by the Canvas renderer closure to decide main-executor sync — one of them is garbage at re-evaluation time.

**Mechanism sketch (for the m16-a design round, NOT root-caused here)**: both faulting closures capture reference objects — `smoother` (plain-class `@State`, read as `smoother.phase` INSIDE the Canvas closure, `VibeMeterView.swift:44`) and `TimelineLanesView.self` (carrying `waveformStore`, `densityStore`, and ~15 context closures, `:40-109`). Wire-driven mutation storms + play-time 60 fps `TimelineView(.animation)` re-render churn the view identity; a canvas display list re-evaluated against a released capture faults. Two independent sites ⇒ fix the PATTERN, not a view: every `Canvas` closure in DAWApp should capture only value snapshots computed OUTSIDE the closure (`VibeMeterView` already documents this contract and violates it by one field). Canvas sites to sweep: `VibeMeterView.swift:43`, `TimelineLanesView.swift:512/:1734/:1853/:2194`, plus any other `Canvas {` in `Sources/DAWApp` (grep). Root-cause verification (is it the @State class? the closure bag? an AttributeGraph race amplified by M15's transport-bar additions?) is the design round's first job — crash 1 vs 2/3 exception types differ (SEGV vs BRK), so the corpse state varies; the capture-discipline fix is right regardless, but the round must prove it kills the class (storm gate below).

**Why now**: the m13/m15 audits ran 53-track storms and repeated `project.new` cycles clean on this machine. M15 added per-frame/layout pressure around the same surfaces (notices chip + count-in stepper in the transport bar, master-strip lane). Regression suspicion recorded honestly as UNPROVEN.

### B2 (F6) — the F-PDC measurement: 113 trials, zero leaks, hazard bounded

**Setup**: tone strip(s) (440 Hz, amp 0.5 clips) with EMPTY chains + one silent dummy strip carrying 16 stacked limiters (chain latency 16 × 240 = 3840 samples) ⇒ every tone strip's compensation ring target = **3840 samples / 80 ms** (snapshot-verified: `compensation histogram {0:1, 3840:40}`; `pdc.trackStageSamples 3840`). If a stale quantum enters a freshly-reset ring at a stop/seek, it must replay as an isolated ~10.7 ms tone blob ~69–80 ms into the post-seam silence — maximally detectable.

**Trials**: (i) 3 plain stops mid-tone (probe1); (ii) 50 seeks tone→silence-region on 2 strips (probe2, corrected `beats` param); (iii) 60 seeks tone→silence on **41 strips** — 40 compensated tone strips playing simultaneously, fattening the render callback so a mid-callback arming store has maximal surface (probe3). Analysis: 1 ms RMS windows, ch0, threshold 0.01, blob detector (active < 30 ms preceded by ≥ 30 ms silence). **Result: zero blobs in all 113 trials; every tone burst edge cut clean at the seam** (`pdc-capture.wav`, `pdc-hammer2.wav`, `pdc-big2.wav` + `analyze-pdc.mjs`).

**Derivation of the bound** (why zero is credible, not luck): the hazard needs the main-actor `armReset` store to land INSIDE one render callback, after the strip's upstream player pull and before its chain-host block — a sub-millisecond window in a 10.7 ms quantum period; unlike the chain-ring case there is no feedback path, so a hit replays once and dies. 113 misses puts per-event p below ~2.6 % (95 % binomial). Offline is structurally safe: manual rendering serializes control and render (the m15-f LIVE-only-double-arm law), and `stopAllPlayers` never fires mid-offline-render. **Disposition: fix as cheap symmetry (m16-f), not as a measured bug.** Gate for the fix: the countdown mirrors `requestReset(passes:)` semantics; offline stays single-armed (null-era byte-identity — the same deviation m15-f proved load-bearing); these three capture probes re-run as the regression gate.

## 4. MIDI CC scope statement (for the m16-b design doc — NOT a design)

What a user with a controller is missing: CC lanes (mod wheel 1, sustain 64, expression 11…), pitch bend, per-clip controller editing in the piano roll, CC capture during recording, CC playback to instruments. Where it must integrate — every structure cited from this tree:

1. **Model** (`DAWCore/Model.swift:103-149`): `MIDINote` is note-only; `Clip` carries `notes: [MIDINote]`. CC/pitch-bend needs a NEW clip-local container (e.g. per-controller point lanes), additive Codable, omit-when-empty — the `masterAutomation` persistence precedent.
2. **The load-bearing boundary**: CC is NOT an `AutomationLane`. `AutomationTarget` (`DAWCore/Automation.swift:18-60`) is the mixer/effect plane (volume/pan/sendLevel/effectParam) applied control-plane at ~60 Hz; CC targets the INSTRUMENT through the sample-accurate event schedule. The design must land CC as clip events on the schedule path; conflating the two would put a mod-wheel ramp on the wrong transport with the wrong timing guarantees.
3. **Schedule** (`DAWEngine/MIDISchedule.swift:5-16`): `ScheduledMIDIEvent` is a 24-byte POD whose comment RESERVES kind 2+ — "CC / pitch bend / sustain arrive additively, post-v0". Controller#/value map onto the existing pitch/velocity bytes; the sort/cursor/timelineID machinery (m14-b append-only extension) carries over unchanged. Render drain: `InstrumentRenderer` must forward kind ≥ 2 to the AU (MusicDeviceMIDIEvent-class call, render-thread-safe — needs the audio-dsp review).
4. **Live input** (`DAWEngine/MIDIInput/`): `MIDIUMPParser` parses ONLY note-on/off (status 0x9/0x8; 0xB/0xE are skip-length cases, `MIDIUMPParser.swift:50-51`); `LiveMIDIEvent` is a 16-byte note POD. CC thru + capture need parser cases, a wider live event ring, and a decimation policy (controllers stream at wire rate). `MIDICaptureSession` pairs on/off by pitch — CC points don't pair; loop-cycle slicing must reuse the m15-b seconds-domain law (`TakeResult.stopBeats`).
5. **UI**: piano roll = `DAWApp/Timeline/ClipMIDIMap.swift` + headless `DAWAppKit/MIDIMapGeometry.swift`. Industry shape: a CC strip under the note grid, per-controller selector. NOT the arrange `AutomationLaneEditor` (that's the mixer plane).
6. **Wire/MCP**: siblings of `clip.setNotes`/`parseNotes` (`Commands.swift:1185+`), snapshot additive keys, MCP tools + catalog rows — the standing triple.

Effort: L (model + engine + capture + UI + wire). Two audits running it has lost on effort-to-payoff; with M15's ergonomics round done, it is the last big model gap between "agent-native DAW" and "DAW a keyboardist can play into" — recommend it as the M16 feature headline behind a design doc with phased landing (playback first, capture second, UI third).
