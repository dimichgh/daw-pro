# Time-stretch & pitch-shift engine evaluation (M5 ii)

Date: 2026-07-05
Scope: choose the library/approach behind offline clip time-stretch and pitch-shift for M5 (ii), with an eye toward later real-time preview (M5+). Licensing is a hard gate — DAW Pro ships commercially.

## Use cases (priority order, per roadmap)

1. **Offline clip stretch/pitch at edit time** — render a stretched copy when the user drags a clip's tempo handle or transposes it, following the M5 (i-b) "bake at schedule time" precedent (`ClipFadeBake.swift`): compute once, cache, swap into the existing player-based schedule.
2. **Real-time preview** — nice-to-have, not required for v0. Scrubbing/auditioning a stretch ratio before committing.
3. **Musical quality on both polyphonic full mixes/stems AND monophonic vocals** — ACE-Step-1.5-generated songs (full mix and/or separated stems, including sung vocal) are a primary source, so vocal/formant fidelity matters as much as mix-bus fidelity.

## Licensing gate — where each candidate lands

- **Hard no standalone**: Rubber Band's GPL track (App Store / proprietary distribution is explicitly disallowed per Breakfast Quay's own license page) — usable only via the paid commercial license.
- **Workable but weakest quality**: SoundTouch (LGPL 2.1). Dynamic-linking a `.dylib` inside a signed `.app` bundle is legally workable (LGPL's obligation is "user can replace the library," satisfiable with an embedded dynamic framework), but this is exactly the scenario where hardened-runtime library validation and notarization add friction (would likely need `com.apple.security.cs.disable-library-validation` or equivalent, a decision better deferred to M9 packaging). Given the quality complaints are the more fundamental disqualifier, not pursuing further regardless of the license workaround.
- **Cleanest**: signalsmith-stretch (MIT — no obligations beyond a copyright notice in documentation/about box).
- **Workable, slightly heavier**: Bungee (MPL 2.0, file-level copyleft — fine as long as we don't hand-edit Bungee's own source files).
- **Zero-risk but capability-limited**: Apple's built-in AUs.

## What the market/forums actually complain about (competitive-edge angle)

- Users of free stretch libraries consistently split into two complaint camps: **"too CPU-heavy for real-time"** (Rubber Band, historically, on mobile/low-power) and **"artifacts on transients/drums"** (SoundTouch). This is exactly why DAW Pro's plan to separate "cheap real-time preview" from "quality offline bake" is the right shape — nobody has to compromise on either axis if we pick different tools for each job.
- Commercial-grade quality (Elastique, Bungee Pro) is reported as meaningfully better specifically on **vocal timbre preservation** — the axis DAW Pro cares about most because of the ACE-Step vocal pipeline. This is the strongest argument for validating vocal quality with an explicit listening test before committing, rather than trusting general-purpose "sounds fine on drums" comparisons.

## Recommendation

**Primary pick: signalsmith-stretch (MIT).** It clears the licensing gate with zero cost and zero ongoing obligations (the cleanest possible answer to a hard commercial-shipping gate), its block-based buffer-in/buffer-out API is a near-exact match for the M5 (i-b) "bake once at schedule/edit time" pattern already proven in this codebase, it ships explicit formant controls needed for the ACE-Step vocal use case, it has credible modern-quality signals (a direct head-to-head preference over Rubber Band in forum testing, production use inside Qt/FFmpeg and audiomentations), and being header-only C++11 makes it the lowest-friction integration of any non-Apple candidate. Its own documentation is honest about weaker spots (formant correction not PSOLA-sharp; time-stretch quality best kept within 0.75×–1.5×), so it should be validated, not assumed, against real ACE-Step vocal stems before this becomes final.

**v0 integration shape**: vendor `signalsmith-stretch.h` + `signalsmith-linear` as a small header-only C++ SPM target (optionally compiled with `SIGNALSMITH_USE_ACCELERATE` to lean on vDSP instead of a bundled FFT), expose it through a thin flat-C shim (opaque handle + create/configure/process/flush/destroy functions — the same interop shape as the existing `CAtomics` C shim precedent, chosen over fighting Swift/C++ template interop), and wrap that in a Swift `OfflineStretcher` facade in `DAWEngine` that the clip-render/bake path calls the same way `ClipFadeBake` calls the fade envelope today.

**Explicit fallback**: if directed listening tests on ACE-Step polyphonic mixes and, especially, monophonic sung-vocal stems show signalsmith falling short (formant artifacts, audible smearing beyond the documented ratio comfort zone), the fallback is **Rubber Band's commercial Standard license (£590 one-time, attribution required — or £1,490 non-attribution while the company has under 10 employees)**. It is the proven quality reference with its own formant-preservation mode, the license cost is trivial against a commercial DAW's budget, and the API shape (`RubberBandStretcher`) is equally offline-friendly. Bungee (MPL 2.0) is worth including in the same listening-test bake-off as a free second opinion, but its vendor's own comparison data suggests the OSS tier is deliberately capped below its paid "Pro" sibling, so it should not be treated as a safer bet than signalsmith going in.

**Do not use**: SoundTouch (quality reputation is the weakest of all candidates for exactly the polyphonic-mix use case we prioritize) as anything but a last-resort, and Rubber Band's GPL track standalone (blocked by the commercial-shipping gate). Apple's `AVAudioUnitTimePitch`/`AVAudioUnitVarispeed` are not the offline bake engine, but are worth keeping in the toolbox for the M5+ real-time preview nice-to-have specifically because DAW Pro already has AU-hosting plumbing (`HostedAUEffect`) that could drive them with near-zero new code — cheap scrub/audition today, replaced by the "real" baked render on commit.

### Actionable takeaways (for the daw-architect / ROADMAP)

1. **ROADMAP M5 (ii)**: adopt signalsmith-stretch as the offline engine; scope a small vocal-focused listening-test spike (ACE-Step output samples, several semitone/tempo ratios) before marking the library choice final, with Rubber Band commercial as the named fallback if it fails.
2. **ARCHITECTURE — integration seam the daw-architect must settle next**:
   - **Where stretched audio lives**: an on-disk render cache (temp file per clip+params, analogous to how fades are baked into copies today) vs. an in-memory buffer owned by the clip. Bears on `.dawproj` persistence — do stretched renders get saved into `media/`, or always regenerated from source + params on load?
   - **Cache key & invalidation**: must key on (source file identity, requested ratio/semitones, sample range, algorithm/library version) and explicitly invalidate on project tempo changes for tempo-synced clips, not just on direct user edits to the clip's own stretch settings.
   - **Ordering with `startOffsetSeconds` and fades (M5 i-a/i-b)**: pin down whether stretch happens before or after the existing fade bake, since stretching changes a clip's frame-length and `fadeInBeats`/`fadeOutBeats` are beat-relative — the beat-to-frame remap has to happen post-stretch, not pre-stretch, or fades will land in the wrong place.
   - **Async job model**: unlike the near-instant fade bake, stretching a full clip is not free — needs a progress/async story over the control protocol (placeholder audio or disabled playback until the render lands) rather than a synchronous main-actor stall, following the same pattern the async AU-instrument-prepare path already established in M3 (vi-a).
   - **Real-time preview boundary (M5+)**: decide whether "preview" means temporarily swapping in `AVAudioUnitTimePitch` during a live scrub versus running signalsmith itself in real time via its `splitComputation` mode — these have very different latency/quality/engineering trade-offs and should be a deliberate choice, not a default.
