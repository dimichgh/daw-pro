# Insert-effect editor UX — what Logic/Cubase do, what DAW Pro should build

**Date:** 2026-07-15
**Trigger:** user report — "In Mix when I add inserts, there is no panel with knobs to adjust any parameters."
**Scope:** built-in effect inserts only (hosted AUs already have `plugin.openUI` floating windows, M3 vi-b — out of scope here except as contrast).

## 0. The gap, confirmed in code

- `InsertRow` (`Sources/DAWApp/Mixer/MixerControls.swift:261`) renders only: bypass dot, name, sidechain KEY picker (compressor/gate only), remove. **No click target opens anything for a built-in effect.** `onOpenWindow` is wired only when `effect.kind == .audioUnit` (`MixerStripView.swift:349`).
- Every built-in kind already has a full parameter schema: `EffectParamSpec.specs(for:)` (`Sources/DAWCore/Effects.swift:467-593`) gives name/range/default/unit for all 9 built-in kinds (`gain, eq, compressor, limiter, reverb, delay, saturator, gate, chorus`; `audioUnit` is the 10th case and correctly returns `[]` — AU params aren't on this surface). This is the exact same table `fx.describe` serializes over the wire (`Commands.swift:918-948`).
- The mutation path already exists and is exactly what the wire uses: `ProjectStore.setEffectParam(trackID:effectID:name:value:)` (`ProjectStore.swift:1958`) and its master-chain twin `setMasterEffectParam(effectID:name:value:)` (`ProjectStore.swift:1414`). `fx.setParam` (`Commands.swift:845-864`) calls these two methods directly — **there is nothing to add on the wire**; a UI panel would call the identical store methods `InsertRow` already calls for bypass/remove.
- Two code comments currently assert something that isn't true: `MixerStripView.swift:29-31` ("the only instrument kind with a plugin window — **built-ins have first-class in-app panels**") and `MixerInsertsSection`'s doc comment (`:311`, "the master target opens no plugin window (**built-ins have in-app editors**)"). Both describe an editor that was never built. The M3 vi-b design doc (`docs/research/design-vi-b-au-windows.md:49-50`) states the same assumption as an explicit non-goal: *"Built-in effects/instruments (kind != .audioUnit) get no windows — they already have first-class in-app panels."* This research confirms that assumption was wrong and names the resulting roadmap item.

## 1. Logic Pro

**Insert add → auto-open.** Adding a plug-in to an insert slot opens its plug-in window immediately by default (toggle: Preferences ▸ Display ▸ Mixer ▸ "Open plug-in window when inserting"). Clicking the center of an already-loaded insert slot re-opens/closes it; the slot itself shows the plug-in name and a bypass LED (blue when active, grey when bypassed) with no parameters visible in the slot. [Apple: Work in the plug-in window](https://support.apple.com/guide/logicpro/work-in-the-plug-in-window-lgcpbc21a1fd/mac); [Route audio via insert effects](https://support.apple.com/en-am/guide/logicpro/lgcp8e838ae3/mac).

**Plug-in window chrome** (from the header, left→right): Link button (chain icon — when on, ONE window follows the selected channel strip and swaps content as you click other channels/slots, so only one plug-in window is ever open at a time in linked mode), Bypass, a Settings/Presets field (click for the presets menu; Prev/Next arrows step through saved settings; Copy/Paste buttons transfer settings between compatible plug-ins), a **Compare** button (turns blue once you've touched a parameter; click to A/B against the setting last saved with the project, click again to return), a View pop-up (**Editor** = the plug-in's native graphical UI; **Controls** = a generic view — every parameter as a horizontal slider with a numeric field, independent of GUI availability — togglable per-plugin or set as the app default), and, on compatible dynamics/gate plug-ins, a **Side Chain** menu (routes any input/aux/track into the detector). [Apple 9.1.6 manual, "Using Common Plug-in Window Functions"](https://help.apple.com/logicpro/mac/9.1.6/en/logicpro/usermanual/chapter_10_section_6.html); [Compare button — Apple Community](https://discussions.apple.com/thread/5296271); [Controls vs Editor view](https://www.logicprohelp.com/forums/topic/139245-switching-plug-ins-view-from-controls-to-editor/).

**Channel-strip thumbnails (no window needed at all).** The channel strip carries a small **Channel EQ display** above the fader: click inserts a Channel EQ (or opens the topmost one if present) and turns the area into a live curve thumbnail; double-click opens the full plug-in window; ⌥-click inserts at the top of the chain; ⇧-click inserts Linear Phase EQ; ⌘-click removes it. [Apple: Use the Channel EQ](https://support.apple.com/guide/logicpro/use-the-channel-eq-lgcpbc219e52/mac). The Compressor has the same idiom: a **gain-reduction meter** sits in the same strip position, togglable between meter/graph display; clicking it inserts/opens the Compressor. [Apple: Compressor overview](https://support.apple.com/guide/logicpro/compressor-overview-lgcef1bec0a5/mac); [dummies.com Compressor/Limiter walkthrough](https://www.dummies.com/article/technology/software/music-recording-software/logic-pro-x/logic-pro-x-plug-ins-compressor-limiter-and-other-dynamic-tools-258846/).

**Takeaway shape:** Logic's model is *two tiers* — an always-visible micro-preview for the ONE most load-bearing insert (EQ curve or GR meter) baked into the channel strip itself, plus a full floating window with standardized chrome (bypass/compare/presets/sidechain/view-toggle) for deep editing. The floating-window part is exactly what generates the "10 plug-in windows open" complaint pattern that shows up in Logic workflow discussions and video tutorials about disabling auto-open — a real cost of the floating-window default that DAW Pro should not import wholesale. [TikTok: "How to stop Logic Pro from automatically opening the plugin window"](https://www.tiktok.com/@sangstersounds/video/7398317160398982442); [Logic Pro Help forum — auto-open regressions](https://www.logicprohelp.com/forums/topic/153347-plugin-windows-dont-open-automatically-in-logic-anymore/).

## 2. Cubase

**Insert slot chrome.** Each of 8 insert slots (6 pre-fader, 2 post-fader) shows a power button (bypass; ⌥-click fully deactivates/frees resources rather than just muting) and an **`e`** (edit) button that opens **Channel Settings** — not a per-plugin window, but one unified editor for the whole channel. [MusicTech Cubase power-user tutorial](https://musictech.com/tutorials/cubase-tutorial-power-user-2/); [Steinberg forums — Alt+click power icon](https://vi-control.net/community/threads/cubase-question.85780/).

**Channel Settings window** is a **rack-based editor** (racks: EQ, Inserts, Strip, Sends — user-selectable via "Select Racks") showing an inline EQ curve, the insert-slot list, and the Channel Strip module rack together, for one channel at a time; it can dock into the **MixConsole's lower zone** or float as its own window, and the whole MixConsole can itself live in a lower zone of the Project window. [Steinberg: Channel Settings](https://www.steinberg.help/r/cubase-pro/15.0/en/cubase_nuendo/topics/mixconsole/mixconsole_using_channel_settings_c.html); [Channel Racks](https://archive.steinberg.help/cubase_pro_artist/v9/en/cubase_nuendo/topics/mixconsole/mixconsole_channel_racks_r.html); [MixConsole in Lower Zone](https://archive.steinberg.help/cubase_pro_artist/v9/en/cubase_nuendo/topics/mixconsole/mixconsole_in_lower_zone_r.html).

**Channel Strip modules vs. insert effects — a real distinction worth stealing.** Insert slots host full VST plug-ins (Steinberg or 3rd-party, any complexity). The **Channel Strip** is a separate, fixed rack of exactly 5 lightweight built-in processors (Gate, Comp, Tools, Sat, Limit) that live in their OWN section of the channel — always in the same slots, always the same modules, pre/post-fader position configurable as one block. [Channel Strip Modules](https://archive.steinberg.help/cubase_pro_artist/v9/en/cubase_nuendo/topics/mixconsole/mixconsole_channel_strip_modules_r.html); [Steinberg forum: strips vs inserts](https://forums.steinberg.net/t/cubase-channel-strips-vs-insert-effects-and-eq-whats-the-difference-and-the-signal-chain/1012138). DAW Pro's built-in kinds (`gain/eq/compressor/limiter/reverb/delay/saturator/gate/chorus`) are already closer to Cubase's "Strip modules" than to arbitrary VSTs — light, fixed-schema, no preset browser needed — which argues for an equally lightweight, always-in-window editor rather than Logic's heavier floating-window-with-presets-and-compare chrome.

**Generic editor fallback** (3rd-party VSTs without/instead of a custom GUI): right-click the plugin title bar → "Switch to Generic Editor" → every parameter as a slider bar, no graphics. [Steinberg forums — generic interface](https://forums.steinberg.net/t/how-to-display-generic-interface-for-vst-plugin/602420). Functionally identical to Logic's Controls view — this generic-slider-list idiom is the correct v1 shape for DAW Pro's built-ins since `EffectParamSpec` is already a flat name/range/default/unit list, not a custom layout.

## 3. Smaller/modern references

**Ableton Live's Device View** is the strongest fit for "radically simpler." A track's devices render in a horizontal strip with each device's own panel showing its actual knobs/sliders inline (native devices always show real controls, not a generic list) — no click-to-open step, no separate window, devices scroll horizontally when the chain is long. Only 3rd-party plug-ins get an *optional* floating window via a "Show/Hide Plug-In Window" toggle; native devices never need one. [Ableton 12 manual: Working with Instruments and Effects](https://www.ableton.com/en/live-manual/12/working-with-instruments-and-effects/); [Instrument/Drum/Effect Racks](https://www.ableton.com/en/manual/instrument-drum-and-effect-racks/).

Live's model doesn't transplant literally — DAW Pro's mixer strips are 132 pt-wide vertical racks in a horizontal scroller (`MixerStripView.swift:73`), not Live's full-width horizontal track lanes, so there's no room for an always-expanded per-device panel with real knobs docked in the strip. The *lesson* that does transplant: **the knobs should be real controls the user drags directly, not a menu/inspector abstraction, and opening them should feel like disclosure, not "launching an app."**

Logic's **Smart Controls** (a handful of mapped macro knobs surfaced per-patch/per-plugin) is the other useful precedent, but it's a curation/mapping system layered on top of full plug-in UIs — worth reusing conceptually for a *future* "curated top params" idea, not for v1.

## 4. Recommendation for DAW Pro

**Primary pattern: in-window Effect Editor card, opened by clicking an `InsertRow`, generic-from-spec in v1, curated per-kind in v2.**

### 4.1 Why not a floating window (Logic) or a Cubase-style docked rack

- DAW Pro already has a firm rule for this exact fork, stated repeatedly in `docs/DESIGN-LANGUAGE.md` for Settings / Instrument Picker / Quantize & Groove / Undo History: a **centered dark-glass modal card over a dimmed scrim, rendered inside the main window content** — explicitly *not* a popover/NSMenu/separate NSWindow, "so it renders inside the main window content and `debug.captureUI` can snapshot it; a popover lives in its own window, invisible to the window cacheDisplay path" (`DESIGN-LANGUAGE.md` m10-n-3 entry). Every one of these four precedents is the same shape: click a chip/row → modal card → close ✕. The built-in Effect Editor should be the fifth instance of this exact pattern, not a new one.
- The floating-window pattern is *already spoken for* — it's the M3 vi-b `plugin.openUI` mechanism, built specifically because a hosted AU needs to render a live, possibly out-of-process `NSView` that cannot be faked generically (`design-vi-b-au-windows.md` §2-§3). Built-in effects have no such requirement: their whole "view" is 2-10 `EffectParamSpec` rows DAW Pro already owns end-to-end. Reusing the floating-window machinery for built-ins would import Logic's window-clutter complaint for zero benefit.
- A docked lower-zone rack (Cubase) doesn't fit today's layout at all — DAW Pro has no lower-zone concept; the Arrange/Mix workspaces are full-panel swaps, not stackable zones. Introducing one is a much bigger structural change than this bug warrants.

### 4.2 Open/close interaction

- **Add a built-in effect** (`+` menu on a strip or master) → the new `InsertRow` appears **and the Effect Editor opens immediately** on it (Logic's auto-open-on-add habit, but scoped to an in-window card, not a new OS window, and with no "disable auto-open" preference needed because closing it is one click).
- **Click anywhere on an existing `InsertRow`** (the name/body area — NOT the bypass dot, NOT the remove context-menu item, which keep their current single-purpose hit targets) toggles the Effect Editor open/closed on that insert.
- **Only one Effect Editor is open at a time**, app-wide (mirrors Logic's "Link" single-window habit, minus the extra chrome needed to control it): opening a second insert's editor replaces the first. This is a deliberate anti-clutter constraint DAW Pro can afford that Logic can't, because there's no "keep window pinned while I edit two plug-ins side by side" use case yet — the mixer strip itself stays the place to compare bypass states across inserts.
- Selecting a different track/bus while an editor is open closes it (the target no longer exists in view) rather than silently retargeting — an explicit, honest state over a magic "Link."

### 4.3 Where it lives

Centered dark-glass modal over a dimmed scrim in the **Mix workspace** (same precedent surface as Settings/Instrument Picker/Quantize/Undo History) — not anchored beside the 132 pt strip (too narrow to anchor a knob-dense card without clipping), not a right-docked panel (that real estate is reserved for the violet AI surfaces — Sketchpad/Copilot/Fix — and an effect editor is emphatically not an AI surface, Rule 3).

### 4.4 Anatomy (v1 — generic-from-spec, ALL 9 built-in kinds)

- Header: kind icon + effect display name + **target label** ("on 'Drums'" / "on Master") + a **BYPASS** chip (mirrors the row's dot, cyan-off/signal-green-on to match the row) + close ✕. No presets/Compare/Link chrome in v1 — DAW Pro's built-ins have no preset system yet (out of scope; flagged below) and "Compare" needs a saved-vs-live diff model that doesn't exist.
- Body: one row per `EffectParamSpec` entry, **in schema-declaration order** (already musically sensible per kind — e.g. EQ walks low-shelf → peak1 → peak2 → high-shelf) — a labeled horizontal slider/knob (reuse the `SendMiniFader`/`VerticalFader` Canvas-drawn idiom rotated horizontal, or a new `ParamSlider` sharing their grab/⌥-fine/double-click-reset gesture contract) with an SF Mono live value + `spec.unit` suffix, double-click resets to `spec.defaultValue`. Every drag calls `store.setEffectParam`/`setMasterEffectParam` directly — **identical to what `fx.setParam` calls**, so UI and wire stay byte-identical by construction (the house rule: "every user-facing operation must be invokable through [the control protocol]" is already satisfied *by the existing method*, nothing new to add there).
- Sidechain: for compressor/gate on a track/bus (not master), the existing `SidechainKeyControl` renders inside the editor too (currently it only lives inline in the collapsed row — keep it in BOTH places, or move it into the editor and leave a compact read-only "KEY ▸ name" chip on the collapsed row; implementer's call, but don't regress the wire-parity property it already has).
- Footer: none needed in v1 (no Compare/presets yet).

### 4.5 Simple vs. Pro density

No new density chip in v1. The entire Inserts section — and therefore every `InsertRow` and the new editor — is **already Pro-gated**: `MixerChannelStrip.body` only renders `MixerInsertsSection` `if isPro` (`MixerStripView.swift:45`). That inheritance is sufficient and consistent with the existing rule against a chip whose two modes coincide ("a toggle that changes nothing is worse than none" — `DESIGN-LANGUAGE.md` "Panels"). v1's editor has exactly one layout, so it earns no chip of its own.

v2 (see below) may introduce a genuine Simple/Pro delta *inside* the editor once curated layouts exist (curated few-knob view vs. full spec list) — at that point, and only then, add a `SimpleProToggle` scoped to the editor (panel id suggestion: `"effectEditor"`, shared across all kinds so it doesn't fragment into nine separate stored prefs) — the same "add the chip only when it's real" rule the Quantize panel followed (grid+strength in both densities, swing/groove Pro-only, added when that distinction became real).

### 4.6 Generic-from-spec vs. curated-per-kind, phased

- **v1 (ship first, closes the reported bug): generic-from-spec for ALL 9 built-in kinds.** One shared `EffectEditorCard` view driven purely by `EffectParamSpec.specs(for: effect.kind)` + the resolved param struct's current values (via a small kind-keyed value-reader, mirroring how `applyEffectParam` is kind-keyed on the store side). This alone fixes the user's complaint for every effect in the app, with no per-kind design work and no curation debt to get wrong.
- **v2 (curated top-3 kinds): EQ, Compressor, Limiter.**
  - **EQ**: replace the flat slider list with a Canvas frequency-response curve (the Channel-EQ-thumbnail idiom) — draggable band nodes for the 2 peaking bands + 2 shelf endpoints, dragging maps directly to `peakNFreq/peakNGainDb/peakNQ` and `lowShelfFreq/lowShelfGainDb`/`highShelfFreq/highShelfGainDb` through the same `setEffectParam` calls; Q as a secondary vertical drag or a small numeric field per node, cyan glow curve (numeric readout, not AI). This is the single highest-leverage curated layout — EQ is the effect users reach for first and a curve is dramatically more legible than 10 sliders.
  - **Compressor**: add a live gain-reduction indicator. **This needs new engine surface** — DAW Pro currently exposes only `mixer.masterAnalysis` (post-master-fader) and per-track level meters (`trackMeters`), no per-effect GR value anywhere in the codebase (confirmed by grep — zero hits for gain-reduction plumbing). Scope this as its own small roadmap item (a `gainReductionDb` field threaded from `CompressorEffect`'s render state through to a pollable per-effect meter, the `trackMeters` pattern) rather than bundling it into the editor's UI work — the editor can ship v2 EQ-curve work without waiting on it, and fall back to the flat slider list for compressor until the meter lands.
  - **Limiter**: smaller lift — a ceiling/release pair, worth a light custom layout (e.g. a two-knob "ceiling + release" row with the same GR-meter dependency as compressor for full effect) but not urgent; could ship v1-generic indefinitely without much cost since it's only 2 params.
  - Reverb/delay/saturator/gate/gain/chorus: stay generic-from-spec — 3-5 flat params each, a curated layout buys little (this mirrors Cubase's own restraint: only EQ and dynamics get dedicated inline widgets; delay/reverb/chorus/saturator stay in the generic Insert/Strip slot list).

### 4.7 Design-language notes for the implementer

- Bypass state: reuse `InsertRow`'s exact semantics — signal-green glow when active, dim when bypassed (never a second color vocabulary for the same state).
- All knobs/sliders: cyan is reserved for the row's OWN accent story (dB readouts, active playback state) per the fader/knob convention already in `MixerControls.swift`; **pan-type or non-signal params stay neutral white** per the existing `PanKnob` precedent (e.g. EQ Q, delay ping-pong toggle-ish param) — only genuinely level/gain-flavored params (dB, gain-linear) take the cyan accent.
- **No violet anywhere in this card** — it is standard mixing chrome, not AI content (Rule 3), same call already made for the Instrument Picker, Quantize panel, and Undo History.
- Ship one `ExplainCatalog` entry per new interactive surface at minimum (`.effectEditor` for the card as a whole is enough per the "one entry per pure-Canvas/gesture surface" precedent used for the arrange clip body and piano-roll grid — no per-parameter entries needed).
- Stage-for-capture: add a `debug.effectEditor {trackId?, effectId?, open?, close?}`-shaped debug command (app-level, debug tier, off `allCommands`/MCP) mirroring `debug.quantizePanel`/`debug.instrumentPicker`, so `debug.captureUI` can snapshot every param state without a live drag.

### 4.8 Wire/store surface — confirmed, nothing new required

- `store.setEffectParam(trackID:effectID:name:value:)` — `Sources/DAWCore/ProjectStore.swift:1958` — exists, throws on unknown track/effect/param, silently clamps to `spec.range`, coalesces per `(track, effect, name)` (so a knob drag is one undo step, same as every other fader in the app).
- `store.setMasterEffectParam(effectID:name:value:)` — `Sources/DAWCore/ProjectStore.swift:1414` — the master-chain twin.
- `fx.setParam` (`Sources/DAWControl/Commands.swift:845-864`) calls **exactly these two methods** — UI and wire are the same store call, so the new panel adds zero new control-protocol commands and zero new MCP tools. The only new wire surface anywhere in this plan is the *optional* v2 `gainReductionDb` metering value (§4.6) and the debug capture-staging command (§4.7), both additive and both off the `fx.*` namespace.
