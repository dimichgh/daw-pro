## 2. New bugs, written to file

### B1 (F1) — CRITICAL: `project.new` crashes the app inside the M9 crash-a fix's own flush path

> **UPDATE (2026-07-12, m13-a design round — supersedes the regression hypothesis below):** a pre-m12-f binary (git-worktree build of HEAD `0ad2e51`, hash-validated as the m12c-era state) **crashes identically on the first routed teardown** — the crash is NOT an m12-f regression; the M9 crash-a fix itself never covered the announce-class teardown shape, and the class also fires on a routed `track.remove` with no `project.new` involved. Full evidence matrix, root-cause model, and the settled fix decision (engine-discard): `docs/research/design-m13a-teardown-crash.md`.

**What happens.** `project.new` over the control protocol SIGSEGVs the app inside `-[AVAudioEngine detachNode:]` → `AVAudioEngineGraph::UpdateGraphAfterReconfig` (`KERN_INVALID_ADDRESS`, garbage pointer — the classic stale-node-list walk). The faulting call is `PlaybackGraph.flushRetiredNodes()` at `PlaybackGraph.swift:412`, invoked from `AudioEngine.tracksDidChange`'s post-rewire restart branch at `AudioEngine.swift:471` — i.e. the detach **against a running engine** that the M9 crash-a fix (`docs/research/fix-teardown-crash.md`) established as the safe path. The fix's core assumption (detach-while-running is always safe) is falsified on the current binary: the crash did not disappear, it moved from "next connect after teardown" into the teardown's own flush.

**Repro (3/3 staging sessions, `.build/debug/DAWApp` built 2026-07-12 09:33, port 17695).**
1. Session 1 (pid 35323): the verbatim ## Blocked recipe — `track.add`×3 (audio + 2 buses), `track.setOutput`→busA, `track.addSend`→busB, `transport.play` 1.2 s, `transport.stop`, `project.new` → **crash during the `project.new` itself** (`DAWApp-2026-07-12-095344.ips`).
2. Session 2 (pid 35488): **no play, no buses, no sends ever** — MIDI + audio clip probes with two intermediate `project.new`s that survived; a later `project.new` (first command of a follow-up script) → same crash (`-095622.ips`).
3. Session 3 (pid 35673): fresh launch, **first wire command** `project.new` 10 s after launch → same crash (`-095844.ips`).

All three stacks are identical frame-for-frame (parsed from the `.ips`; copies in scratchpad `m13-audit/`). The trigger is timing/heap-reuse dependent (session 2 survived two `project.new`s before dying on the third), which is how it slipped through the m12-f/g live gates run earlier the same day.

**Why this is (very likely) an m12-f-era regression, not a never-fixed bug.** The fix was verified live 3× on 2026-07-10 (`repro-teardown-crash.md:36`). m12-f (2026-07-12) then changed `PlaybackGraph.swift` — one of its declared 6 render-path files, including "RoutingKey.sidechainEdges + reconcile + **teardown (retire discipline)**" (`ROADMAP.md:207`) — and the m12-f/g gates were offline-render/wire-mutation focused; no play→`project.new`→mutate crash campaign was re-run. Diagnostic breadcrumbs for the fixer: sessions 2/3 had no bus-facing wiring of mine, so either the announce (`willMutateRoutingTopology`, fired once-per-pass at `PlaybackGraph.swift:492-499`) now triggers for plain strips, or the retire bin held nodes from the startup session's teardown across a window the m12-f teardown changes widened. Note also `engine.reset()` before `start()` in the restart branch (`AudioEngine.swift:465`) — the M9 evidence matrix showed `reset()` does NOT purge AVFoundation's pending detach state (experiment D1, `fix-teardown-crash.md:34`).

**Impact.** Any user doing File→New (or any agent doing `project.new`) after the engine has run loses the app. Data risk bounded by autosave/recovery, but this is a beta-blocking crash class.

### B2 (F2) — HIGH: four mutation paths silently violate the no-silent-overlap invariant

m11-d settled the invariant "no SILENT overlap of ordinary same-track clips" and implemented it for `clip.move` + the ⌘I import batch, fixing the measured +6.02 dB doubling. These four paths bypass it (all live-proven on 17695, scratchpad `probe.mjs`/`probe3.mjs` output):

1. **`clip.trim` extend** — `trimClip` clamps only to `minClipLengthBeats` (`ProjectStore.swift:2221`), so extending a clip's window over its neighbor commits an overlap. Observed: clips `[0,124]`+`[124,244]` → trim c1 to length 125 → overlap `[124,125]`, both clips intact.
2. **`clip.stretchToLength`** — `stretchClip` retargets `lengthBeats` (`:2646-2691`) with no resolution. Observed overlap `[124,126]`.
3. **`clip.addAudio atBeat`** — `importAudio` (`:1767-1799`) lands a clip at any beat with no resolution (the batch path resolves; the single-clip wire verb does not). Observed: second import at beat 1 over a 120-beat resident → overlap `[1,120]` — a full-length silent doubling, the worst shape of the class.
4. **`clip.addMIDI atBeat`** — `addMIDIClip` (`:1811-1854`) appends anywhere. Observed overlap `[2,4]`; consequence is double-note scheduling rather than +6 dB, but it's the same invariant breach (both clips' notes play).

Fix shape: route all four commits through the `resolveOverlap` policy (`:2379-2396`) exactly as `moveClip` does (active clip wins, resident trims/removes inside the SAME `performEdit`, take/comp members exempt via the `requireNotCompMember` precedent).

### B3 (F3) — MED-HIGH: routing topology edits mid-recording break the rolling take

`track.addSend`/`track.removeSend`/`track.setOutput`/`track.remove`/`fx.setSidechain` have no `isRecording` guard (contrast `setPunch` `ProjectStore.swift:762`, `setMetronome` `:790`, `setTempoMap` `:388` — all refuse with `transportBusy` teaching errors). When one of these announces a routing rewire, the quiesce hook (`AudioEngine.swift:272-285`) winds playback down and **stops the engine** with `activeTake` untouched; the resume path (`:457-514`) restores playback via a fresh anchor but never re-arms or finalizes the take — the writer's host-time alignment references the dead anchor. Expected symptoms: truncated/misaligned take while the store still reports `isRecording: true`. Reachable today by any agent (Copilot included) editing routing while the user records. Code-proven; the live repro (input-device recording + mid-take `track.addSend`) belongs to the fix item's gate.

## 5. Actionable takeaways

- **m13-a is not optional.** Three crash reports in one audit session on the current binary, one of them on the app's first wire command. Until it lands, every live gate that calls `project.new` more than once is rolling dice — including our own verification scripts.
- The overlap family (m13-b) shows the m11-d invariant was landed verb-by-verb instead of at the commit choke point; the fix should leave ONE policy site every future geometry verb flows through, so the class dies permanently.
- m13-d (master chain) is the highest product-value feature on the list by the vision's own words, and its PDC hook already exists — the engine shape is the proven strip sandwich, not new architecture.
- The m10-q pattern repeated: the biggest wins were things the wire already did that no human could reach (F6), and the worst bugs were on the newest and the oldest seams, not the middle.
