```
┌────────────────────────────────────────────────────────────┐
│ DAWApp (SwiftUI)                                           │
│   Views ── observe ──► ProjectStore (@MainActor)           │
│                            │ commands                      │
│                            ▼                               │
│                       DAWCore (pure domain model)          │
│                            │ engine ops                    │
│                            ▼                               │
│                       DAWEngine (AVAudioEngine/CoreAudio)  │
│                        · playback graph  · metering        │
│                        · AU hosting      · offline render  │
└────────────▲───────────────────────────────────────────────┘
             │ same command surface as the UI
     DAWControl WebSocket server (127.0.0.1:17600, JSON)
             ▲
             │ ws bridge
     mcp-server (TypeScript, stdio MCP) ◄── Claude / any MCP client
             │
             └── direct HTTPS: Anthropic · OpenAI · Suno · GPT Image
```

## Principles

1. **One command surface.** UI actions and control-protocol commands converge on the same `ProjectStore` methods. If a feature works in the UI but has no command, it is unfinished (see CLAUDE.md conventions).
2. **The render thread is sacred.** `DAWEngine` communicates with the render path via lock-free/atomic state and preallocated buffers. Metering flows out via periodic taps, never blocking.
3. **DAWCore is headless.** The whole domain model builds and tests without AVFoundation or SwiftUI. This is what makes autonomous agent development fast and safe.
4. **MCP server is thin.** It translates MCP tools → control commands, and hosts the AI-provider calls (lyrics, Suno, images). It holds no DAW state of its own; `project_snapshot` always reflects the app.

## RT-safety invariants (audited 2026-07-10, M9 perf-a — full inventory: docs/research/audit-rt-safety.md)

Two tiers. Every new engine change is reviewed against this section.

**Tier 1 — render thread (strict).** The AVAudioSourceNode render block (`InstrumentRenderer.renderQuantum`), `ChainHostAU.internalRenderBlock`, everything they synchronously call (instruments, effect walks, PDC ring, automation stages), and the CoreMIDI receive block (realtime thread — same rules). Forbidden: heap allocation, locks/semaphores, actor hops or `Task` creation, ObjC dynamic dispatch beyond the callback entry itself, file I/O, logging, unbounded loops, ObjC exceptions. Allowed: preallocated buffers, `CAtomics`, pure math (libm), bounded loops over preallocated storage, Swift protocol witness dispatch, and calls into an AU's captured `renderBlock`/`scheduleMIDIEventBlock`.

**Tier 2 — tap/queue side (relaxed, documented).** AVFoundation tap closures (master + per-strip meters, input capture), notification blocks, and `RecordingWriter`'s private serial queue run on dispatch queues, never the render thread. Allocation is tolerated but must be minimal and deliberate; the canonical publish idiom is: compute scalars in the tap, hop once via `Task { @MainActor }` with a Sendable value. Keep `@Sendable` on every AVFoundation/CoreMIDI callback closure (Swift 6 trap: an inherited-main-actor closure traps when invoked on the callback's thread).

**State crossing the Tier-1 boundary** uses exactly three shapes:
- *Pointer publish (main → render)*: main actor `passRetained` → `daw_atomic_ptr` exchange; the slot holds +1; the displaced object goes to a MAIN-ACTOR retire bin released only when > 1 s old; the render side borrows via `takeUnretainedValue` and never retains. (ARC may still emit retain/release around the borrow — lock-free and non-allocating; the retire bin + slot retain make a render-thread LAST release impossible. Every new publish site copies this pattern exactly.)
- *Scalar flags/targets (main → render)*: `daw_atomic_u32` — store-release from the main actor, load-acquire or exchange-acq_rel (consume-once flags) render-side. CAtomics structs are ALWAYS heap-allocated (`UnsafeMutablePointer.allocate`) for stable addresses — never `&property`.
- *Multi-producer accumulators (render → reader)*: monotone `daw_atomic_u64` counters bumped from render callbacks via `daw_atomic_u64_add` (fetch_add, acq_rel — a single LDADD on arm64) and monotone peaks via `daw_atomic_u64_max` (bounded CAS-max; boundedness argument in CAtomics.h). EXACT under any number of concurrent producers. The PROVEN overlap topology (perf-d addendum, docs/research/perf-c-load-profile.md) is a live graph's render thread plus concurrent OFFLINE manual renders sharing one context — in-block time exceeded wall time during perf-c's offline windows; the live graph itself measured single-threaded here (pre/post-fix counts identical at 16 debug tracks). The shape assumes nothing about producer count, so any future parallel-render topology stays exact. One reader at a time derives reset windows by baseline subtraction; each indivisible add lands in exactly one window. Canonical instance: `EnginePerformanceContext` (M9 perf-d). Same heap-allocation rule as all CAtomics structs.

**Documented exceptions (the complete list — additions need a design decision here):**
1. `MasterMixAnalyzer.snapshot()` — one 24-element bands-array allocation per snapshot, tap-queue side (vm-a precedent).
2. `HostedAUEffect.reset()` → `auAudioUnit.reset()` — one ObjC message on the render thread, EDGE-TRIGGERED only (stop / un-bypass); effects have no MIDI all-notes-off path. Must never become per-quantum.
3. `RecordingWriter.receivedAudio` (`OSAllocatedUnfairLock`) — taken on the INPUT TAP queue per delivery so the no-input watchdog can read cross-thread; this pattern is Tier-2-only and must never migrate into a render callback.
4. `MIDIInputRTContext` event/overflow counters — non-atomic load→store increments, valid ONLY under the one-UMP-port single-producer rule.
5. `EnginePerformanceContext` EMA + last-seen cells (`recentLoadBits`, `lastQuantumFrames`, `lastSampleRateHz`) — multi-producer racy-benign load→store on atomic u64 cells: concurrent producers (live render thread + offline manual renders) can lose an EMA step or interleave the last-seen pair, a bounded, decaying distortion of ballistic display values — never a torn value (C11 atomics), never corruption. The five monotone accumulators and the peak are NOT covered here: since M9 perf-d they use the sanctioned multi-producer accumulator shape above (the "rare overlap, lost increment tolerable" tolerance was retired: an in-process 8-thread falsification measured 45 % loss on the old load→store RMW, and live+offline overlap is real; the live 43 %-of-expected callback rate at 16 debug tracks turned out to be REAL dropped device IO cycles under debug saturation — which exact counters now expose honestly as a wire-visible saturation signal; perf-d addendum in docs/research/perf-c-load-profile.md). Telemetry only; this tolerance must never migrate to state that steers rendering.

**Boundary discipline:** routing rewires never mutate under a running engine — since m13-a an announce-class pass (`willMutateRoutingTopology`) on a once-rendered engine ABORTS reconcile and `AudioEngine.rebuildEngine(reason:)` discards the WHOLE `AVAudioEngine` + `PlaybackGraph` and cold-builds from the model (the hook only winds active playback down; project boundaries force the same rebuild unconditionally via `projectWillReplace()`). Teardown detaches of ONCE-RENDERED nodes execute only against a CONTINUOUSLY-RUNNING engine (`teardownDetach(_:)` is the only sanctioned teardown detach); a once-rendered STOPPED engine is never surgically detached — nodes stay attached and the graph flags `needsEngineRebuild` (the M9 retire-bin/flush discipline was FALSIFIED 2026-07-12: nodes that rendered and then sat attached across a stop→reset→start boundary are already stale in AVFoundation's internal node vector, and the post-restart flush-detach itself faulted — six identical `.ips` across two binaries; docs/research/design-m13a-teardown-crash.md supersedes fix-teardown-crash.md); `CompensationDelayState.allocate/release` runs only inside `(de)allocateRenderResources` / strip init (host guarantees no concurrent render); player/click scheduling, fade bakes, and all file I/O are main-actor or worker-queue control-plane; the stretch engine (`CSignalsmithStretch`) is worker-task-only — the live graph only swaps file URLs at reconcile. The M9 crash-c engine watchdog is READER-SIDE only (its heartbeat is one load-acquire on the perf-b telemetry's lifetime callback cell, its tick loop lives on the engine's actor, and its restart path IS the config-change recovery — `recoverEngine()` — so it adds zero render-thread code and no new recovery lifecycle). Audit verdict 2026-07-10: 0 violations, 6 watch items (see the audit doc).

**Recording guard for routing topology (m13-c):** because an announce-class rewire on a once-rendered engine discards and cold-rebuilds the whole engine (above), the store REFUSES every announce-capable routing mutation while `transport.isRecording` — it would tear the engine out from under the in-flight take, truncating/misaligning the capture while `isRecording` still reads true. The single choke point is `ProjectStore.requireRoutingMutationAllowed(_:)` (@MainActor, below both the wire and UI entry), which throws `transportBusy` ("cannot … while recording — stop first" — the `setPunch`/`setTempoMap` precedent). Guarded unconditionally: `track.addSend`/`removeSend`/`setOutput`, `track.remove`, `fx.setSidechain`. Guarded only when the specific edit is announce-capable: `fx.remove` (removing a *keyed* effect clears a key edge), `track.setInstrument` (an instrument-NODE rebuild on a ROUTED strip). PARAM-ONLY edits stay legal mid-take — volume/pan/mute/solo, send LEVEL (`track.setSend`), `fx.setParam`/`setBypass`, clip edits, transport — as do the attach-only `track.add`/`fx.add`/`fx.reorder`, which never announce.

## Control protocol (v0)

Request: `{"id": "…", "command": "transport.play", "params": {…}}`
Response: `{"id": "…", "ok": true, "result": {…}}` or `{"id": "…", "ok": false, "error": "…"}`

Command namespaces: `transport.*`, `track.*`, `clip.*`, `take.*`, `groove.*`, `mixer.*`, `engine.*`, `project.*`, `render.*`, `ai.*`, `plugin.*`, `input.*`, `midi.*`, `arrange.*` (m15-d adds 3 arrangement commands: `insertBars`, `deleteBars`, and internal state tracking via the existing clip-mutation flow). **125 commands** verified against `Sources/DAWControl/Commands.swift` (m15-d +3; m16-b2 +2 controller-lane verbs). The canonical list lives in `Sources/DAWControl/Commands.swift` and must stay in sync with the MCP tools registered in `mcp-server/src/server.ts` (`src/index.ts` is only the stdio entry point — it imports `server` from `server.ts` and connects a transport; it registers no tools itself).

**Parity + schema-richness is test-enforced.** `npm test` in `mcp-server/` (`mcp-server/test/audit-tools.test.ts`, node's built-in test runner, no new deps) spins up the real `McpServer` from `server.ts` on an in-memory transport pair, drives it with a real MCP SDK `Client` (`tools/list` only — no network, no running app required), and cross-checks the result against `Command.allCommands` parsed out of `Sources/DAWControl/Commands.swift`. It asserts: every command maps to exactly one tool and vice versa (bijection, mechanical name rule "`.`→`_`", camelCase→snake_case, e.g. `clip.addAudio`→`clip_add_audio`), every tool has a non-empty title and a beginner-readable description (≥40 chars), every `inputSchema` property has a non-empty description (recursively, including nested objects/array items), and no tool name has a capital letter or a dot. Two exception tables cover intentional deviations from the mechanical rule:
- **Table A** — commands whose tool name the mechanical rule can't produce, so an explicit override is pinned. Six generation commands historically drop the `ai_` prefix (live in user configs, not renamed): `ai.generateSong`→`generate_song`, `ai.extractStems`→`extract_stems`, `ai.legoGenerate`→`lego_generate`, `ai.generationStatus`→`generation_status`, `ai.importGeneration`→`import_generation`, `ai.importGeneratedStems`→`import_generated_stems`. Plus the acronym-plural case `plugin.listOpenUIs`→`plugin_list_open_uis` (the mechanical camelCase→snake_case rule mis-splits the "UIs" run into `plugin_list_open_u_is`, so the readable name is pinned). All other `ai.*` commands keep the prefix (`ai_write_lyrics`, `ai_provider_status`, `ai_sidecar_*`, `ai_repaint_audio`, `ai_fix_clip_region`, `ai_import_clip_fix`, `ai_copilot_*`, …).
- **Table B** — 3 tools with no backing command by design, since they call an AI provider's API directly from the MCP server rather than bridging to the app: `generate_image`, `generate_lyrics`, `generate_song_suno`.

**Integration suite** (`mcp-server/test/integration.test.ts`, joined into `npm test` alongside the audit) drives the REAL wire end-to-end instead of the audit's in-memory transport: it spawns the actual `DAWApp` binary (`DAWPRO_APP_BINARY` env override, default `.build/debug/DAWApp`, loopback port 17690 — SKIPS the whole file with a clear message if no binary is found, never a silent pass) and a REAL SDK `Client` over `StdioClientTransport` launching `node dist/index.js`, so a call genuinely crosses client → stdio → mcp-server → control WebSocket → app and back. ~12 tests cover tool-count parity, the core project/track/clip round trip, all three composition macros over real MCP, `edit.undo`'s one-step scaffold revert, and the error surface — opening with regression pins for two bugs the suite's own first real-transport run surfaced: (1) `textResult`'s **void-result contract** — a command that resolves no result body (`track.setVolume`, `transport.play`, …) reads back as the literal string `"ok"`, not `JSON.stringify(undefined)` (the bare JS value `undefined`), which used to produce an invalid `{type:"text", text: undefined}` content item that the CLIENT's own result-schema validation rejected with `-32602` even though the app-side command had already executed; and (2) `DawBridge`'s **per-command timeout** — `render.bounce`/`render.mixdown`/`render.stems`/`render.measureLoudness` get 180s (`LONG_RUNNING_COMMANDS`) instead of the flat 5s default, since a full-length render/loudness pass can take tens of seconds of wall time and the reply used to be discarded (and lost) the moment the default timeout fired, even though the app finished the work moments later. Teardown (client close, then app SIGTERM escalating to SIGKILL) runs in `try`/`finally` at every level so a failure never leaves a stray `DAWApp` process behind.

`take.*` (M5 iii-b) — comping control surface over the iii-a take/comp domain (`Sources/DAWCore/Takes.swift`, `ProjectStore+Takes.swift`): `take.group` (form a group from >= 2 overlapping clips), `take.setComp`/`take.select` (replace/swap the comp — which lane plays which absolute-beat range), `take.removeLane` (delete an unused take), `take.flatten` (dissolve the group back to ordinary, editable clips), `take.move` (reposition the whole group), `take.setCrossfade` (join crossfade width in seconds), `take.autoAlign` (M6 v-d — onset-based micro-alignment of a take to the group's reference lane; own paragraph below). Every response threads the model's own Codable (`TakeGroup`/`CompSegment`/member `Clip`s) onto the wire so it can never drift from persistence; `take.setComp`'s `segments` param round-trips through `CompSegment`'s Codable with per-index errors. Member clips (`Clip.takeGroupID != nil`) reject the ordinary clip-edit commands with `clipInTakeGroup`, pointing at `take.setComp`/`take.flatten`. Recording a take that overlaps an existing take/group on its track auto-groups (`ProjectStore.autoGroupRecordedTake`, called from `finishTake`, audio-only in v0; newest take wins the default comp) — one `performEdit` covers the whole grouped take, so it is still a single undo step. Mirrored 1:1 by 8 MCP tools (`take_group`, `take_set_comp`, `take_select`, `take_remove_lane`, `take_flatten`, `take_move`, `take_set_crossfade`, `take_auto_align`) in `mcp-server/src/server.ts`.

`take.autoAlign` (M6 v-d) — onset-based micro-alignment of take lanes (`Sources/DAWCore/TakeAlignment.swift`, `ProjectStore+TakeAlignment.swift`). AI-generated takes (the "AI Fix N" lanes `ai.importClipFix` lands) can phrase a hair early/late versus the recorded performance; this command measures the micro-offset between a take lane and the group's FIRST lane (the reference — AI Fix groups are built reference-first) and nudges the take by −offset so its onsets lock on. Onset detection REUSES the one shared engine detector (`detectTransients`, iii-e — content-key cached, never duplicated DSP) on both lanes' files over the OVERLAP of their spans; the pure `TakeAligner` grid-searches ±`searchWindowMs` (10–500, default 150) at 1 ms steps with greedy nearest matching (25 ms tolerance, each reference onset consumable once), tie-breaks deterministically (most matches → smallest mean deviation → smallest |offset|), then refines the winner to the MEDIAN of matched pair deltas — sub-millisecond, no grid quantization. Fewer than 2 matches → `alignmentInconclusive` (counts + advice: widen the window or `take.move` manually) — the aligner never guesses. `apply` (default true) moves the lane inside ONE `performEdit("Align Take")` on the same lane-startBeat + member-rebuild machinery `take.move` uses; an apply whose earlier-move exceeds the lane's headroom before beat 0 throws `alignmentWouldCrossTimelineStart` (required move + headroom + `take.move` advice) instead of silently clamping — `applied: true` always means the take now sits aligned; `apply: false` is a pure dry-run. Response = the report `{offsetMs, offsetBeats, matchedOnsets, referenceOnsets, candidateOnsets, confidence, applied}` (positive offset = the take was late; `confidence = matched / max(1, min(ref, cand))`). Mirrored by the `take_auto_align` MCP tool.

`clip.detectTransients` (M5 iii-e) — offline transient analysis surface over the engine's `detectTransients(inFileAt:sensitivity:)` (spectral-flux `TransientAnalyzer` + content-keyed `TransientCache` JSON sidecars in `Sources/DAWEngine/Analysis/`; the only engine addition in the whole M5 iii epic, never on the render thread). Read-only and pull-only: markers are SOURCE-FILE seconds (geometry-free — trims/splits never invalidate the cache), the response `{transients: [{sourceSeconds, beat, strength}], count}` window-filters to the clip and maps beats at the current tempo/stretch, nothing is persisted or snapshotted. MIDI clips reject verbatim (`transientsRequireAudioClip`). Mirrored by the `clip_detect_transients` MCP tool; feeds iii-f `clip.quantizeAudio` and iii-g groove extraction.

`clip.quantizeAudio` (M5 iii-f) — destructive audio quantize over the pure `AudioQuantizePlan` (`Sources/DAWCore/AudioQuantize.swift`): detect transients (iii-e path) → slice the clip at onsets → monotone strength-lerp nudge of each slice to `QuantizeTarget.nearest` (same grid/strength/swing evaluator as MIDI `clip.quantize`) → replace the clip with the slice clips in ONE `performEdit("Quantize Audio")` (undo restores the single original clip, id included). Slices are NATURAL-LENGTH — each reads only its own `[onsetᵢ, onsetᵢ₊₁)` source span, never a neighbour's attack (deviation from the settled design's source-continuation gap-fill, which measurably doubled transients on the rendered bounce) — with representational equal-power crossfades at compressed joins (CompFlattener mechanism) and clean-cut silence at expanded gaps. No per-slice stretch in v0 (elastic v1 flagged); params `{gridBeats, strength, swingPercent, sensitivity, crossfadeMs 0–50, minSliceBeats}`. Rejections verbatim: MIDI (`quantizeRequiresAudioClip`), non-identity stretch (`audioQuantizeStretchUnsupported`), comp member (`clipInTakeGroup`), fewer than 2 usable transients (`audioQuantizeNoTransients`). Mirrored by the `clip_quantize_audio` MCP tool; the `groove` param (iii-g) is live in both layers.

`clip.humanize` (M7 macro-a) — seeded "human feel" jitter for a MIDI clip, the inverse-spirited sibling of `clip.quantize` (`Sources/DAWCore/ProjectStore+Humanize.swift`, `humanizeClipNotes(clipID:timingBeats:velocityRange:seed:)`). Where quantize pulls onsets TOWARD the grid, humanize nudges each note OFF it: an independent uniform timing offset in `[-timingBeats, +timingBeats]` beats (`timingBeats` 0…0.25, default 0.02) and an independent uniform integer velocity offset in `[-velocityRange, +velocityRange]` (`velocityRange` 0…64, default 8), applied as ONE `performEdit("Humanize")` on the same `tracksDidChange` restart seam `setClipNotes`/`quantizeClipNotes` ride (the engine never learns about humanize). Onsets clamp inside the clip (clip-local 0 … `lengthBeats − minLengthBeats`, never at/after the end), velocities clamp to 1…127; note LENGTHS, ids, and ARRAY ORDER are all preserved (unlike quantize, this does NOT re-canonicalize — index `i` in maps 1:1 to index `i` out so an agent can correlate the jittered notes with what it sent). DETERMINISM is the contract: a private splitmix64 `SeededRandomNumberGenerator` (Swift's system RNG is not seedable) drives all draws, so the same `seed` yields byte-identical notes; `seed == nil` draws a fresh seed internally in `[0, 2^53)` (Double-exact so it survives the JSON number wire) and RETURNS it as `seedUsed` alongside the updated clip fields — replay that value (or omit `seed` again) to reproduce or re-roll a take. `timingBeats == 0`/`velocityRange == 0` are valid no-ops on that axis (the draw is skipped, not consumed); both zero round-trips the clip bit-identically. NO coalescing — a re-run compounds the jitter (undo first, or reuse the seed). MIDI CLIPS ONLY: an audio clip rejects verbatim (`notAMIDIClip`, the `clip.setNotes` mapping); a comp MEMBER rejects with `clipInTakeGroup` via the shared `requireNotCompMember` guard (edit the comp or `take.flatten` first — the same store invariant quantize/setNotes enforce); unknown id → `clipNotFound`. Mirrored 1:1 by the `clip_humanize` MCP tool (`mcp-server/src/server.ts`; 93→94) whose schema documents the determinism/seedUsed re-roll workflow.

`mixer.applyPreset` (M7 macro-b) — applies a named, curated mixer preset to one track/bus strip, over `ProjectStore.applyMixerPreset(trackID:presetName:)` (`Sources/DAWCore/MixerPresets.swift`). The preset's built-in-effect insert chain REPLACES the strip's current chain wholesale in ONE `performEdit("Apply Preset '<DisplayName>'")` — a single `edit.undo` restores the exact prior chain (ids and order included) — while volume/pan/sends are untouched (presets are TONE, not balance); it works on audio, instrument, AND bus tracks. The catalog is a static, versioned, code-defined table (`MixerPresetCatalog.v1`, the `CopilotToolCatalog` precedent): each `MixerPreset {name (kebab-case id), displayName, summary, chain: [EffectDescriptor]}` maps a musical intent onto the REAL built-in param surfaces using only the FX pack's EQ / compressor / limiter / gain — six presets v1: `drum-bus-glue`, `vocal-presence`, `bass-tight`, `master-glue`, `warm-keys`, `clean-boost`. Chain templates carry throwaway ids; each apply mints FRESH effect ids (`freshChain()`) so two strips holding the same preset still address distinct effects via `fx.*`. Because the built-in EQ has no dedicated high-pass, every "rumble/sub trim" intent is a low-shelf with negative gain — the closest available shape, documented in the catalog comment, no new effect type added (the one deliberate approximation). Params: `trackId` + `preset` (required, one of the six names); an unknown preset throws `mixerPresetNotFound` LISTING every valid name (the `unknownEffectParam` precedent), `trackNotFound` via the existing mapping. Response is the shared fx-mutation shape `{trackId, effects}` (the resolved new chain, in order) so the agent sees exactly what the preset laid down. Mirrored 1:1 by the `mixer_apply_preset` MCP tool (`mcp-server/src/server.ts`; 94→95) whose `preset` enum describes all six with one-line summaries and whose description carries the replace-wholesale / one-undo-step / tone-not-balance semantics. Proven by DAWCore `MixerPresetTests` (catalog integrity exercised by APPLYING every preset, replace-existing-chain, undo-restores-exact-prior-chain, all three track kinds, volume/pan untouched, unknown-preset lists all names) and DAWControl `MixerPresetCommandTests` (routing, field-named validation, unknown-preset name listing, happy-path chain-in-order).

`mixer.masterAnalysis` (M8 vm-a) — the session vibe meter's data source: the latest master-mix analysis snapshot, read-only, no params, over `ProjectStore.masterAnalysis()` → `AudioEngineControlling.masterAnalysis()` (poll-based like meters; the floor snapshot when headless). The DSP core is `MasterMixAnalyzer` (`Sources/DAWEngine/Analysis/MasterMixAnalyzer.swift`): a preallocated, allocation-free-after-init unit — 2048-sample Hann frames at 1024 hop, vDSP packed real FFT — fed from INSIDE the existing master meter tap on `mainMixerNode` (one tap per bus is an AVFoundation limit; the tap is post-master-fader, so what you hear is what is analyzed). The analyzer's DSP state is touched only on AVFAudio's serial tap queue; each produced snapshot VALUE rides the same `Task { @MainActor }` hop the `MeterFrame` publish already uses (~30–60 Hz while the engine runs) into a main-actor cache the app (vm-b) and this command poll. Wire shape `{bands: [24 dB values], levelDB, peakDB, centroidHz, flux}` — `bands` are the 24 geometric subdivisions of 40 Hz → 16 kHz (floor −80 dB; a band narrower than one FFT bin reads its nearest bin), `levelDB` is short-term RMS in dB (asymmetric one-pole ballistics; NOT LUFS — `render.measureLoudness` keeps EBU gating), `peakDB` a held peak with −20 dB/s release, `centroidHz` the spectral centroid ("brightness", 0 when silent), `flux` normalized half-wave-rectified spectral flux 0–1 ("energy movement", instant attack / 0.75×-per-frame release). Every field is FINITE by contract: floor-clamped dB, NaN/Inf-poisoned input frames are skipped whole, and a stopped/silent session DECAYS to the floors (`MasterAnalysisSnapshot.floor`) — never an error. The snapshot type lives in DAWCore (`Model.swift`, beside `MeterFrame`) because the engine protocol surface is DAWCore. Mirrored 1:1 by the `mixer_master_analysis` MCP tool (`mcp-server/src/server.ts`; 96→97). Proven by DAWEngine `MasterMixAnalyzerTests` (1 kHz sine centroid/band-peak/RMS-±1.5 dB, exact silence floors, flux-on-spectral-change, decay-to-floor, poisoned-input JSON round-trip, chunk-size invariance, stereo mono-mix parity) and DAWControl `MasterAnalysisCommandTests` (wire shape, headless floors, engine snapshot threading).

`engine.performanceStats` (M9 perf-b) — render-load / overrun telemetry, the first `engine.*` command and the measurement seam perf-c's load-profiling pass reads. Params `{reset?: bool, default false}`; response = `EnginePerformanceStats` verbatim: `{callbackCount, renderedFrames, renderTimeNs, peakCallbackNs, overrunCount, averageLoad, recentLoad, sampleRate, quantumFrames, sinceResetSeconds}`, over `ProjectStore.performanceStats(reset:)` → `AudioEngineControlling.performanceStats(reset:)` (`.idle` all-zeros when headless; optional-capability default). The RT core is `EnginePerformanceContext` (`Sources/DAWEngine/EnginePerformance.swift`): one heap allocation of `daw_atomic_u64` cells (the CAtomics stable-address idiom; the header carries u64 load-acquire/store-release plus the perf-d multi-producer ops `daw_atomic_u64_add` / `daw_atomic_u64_max`) stamped by BOTH Tier-1 workhorses — `InstrumentRenderer.renderQuantum` (entry/exit `mach_absolute_time()`, a commpage read, on every return path) and `ChainHostAU.internalRenderBlock` (entry AFTER `pullInputBlock` returns so nested upstream renders are never double-counted; wired via `setPerformanceContext` at node creation, a `PerformanceBox` following the Scratch write-window discipline) — so the sum ≈ the engine's own DSP work (AVFoundation internals — player file reads, mixer SRC/sum — are outside the measurement, and callbackCount/renderedFrames scale with strip count). Render-side math is integer-only (timebase numer/denom precomputed at init; all divisors guarded ≥ 1) plus one non-trapping Double for the `recentLoad` one-pole EMA (α = 0.0106 ≈ 1 − exp(−1/93.75): ~1 s time constant at 512-frame quanta at 48 kHz per block; N blocks shorten it to ~1/N s); increments use the sanctioned multi-producer accumulator shape (M9 perf-d): `daw_atomic_u64_add` fetch_add on the five monotone cells and `daw_atomic_u64_max` CAS-max on the peak — exact under the parallel render pool — while the EMA/last-seen cells stay racy-benign (invariants exception 5). `overrunCount` is a BUDGET-OVERRUN PROXY, not a CoreAudio xrun observation (AVAudioEngine exposes no xrun count): a callback whose elapsed strictly exceeded its own quantum budget (frames/sampleRate) — one block alone eating a whole quantum, already trouble. `averageLoad` = windowed renderTimeNs / accumulated per-callback budget, derived READER-side only. Offline renders COUNT: `AudioEngine.renderOffline` hands its context to the `OfflineRenderer` → `PlaybackGraph`, which is what makes headless bounces profilable (and the E2E test possible). `engine.watchdogStatus` (M9 crash-c) — the engine watchdog's state read: the stall detector that keeps the engine from dying SILENTLY. Known accepted imprecision (documented, not fixed): a concurrent offline render bumps the same shared context, which can only DELAY detection — never false-positive. `ai.fixClipRegion` / `ai.importClipFix` (M6 v-b) — the clip vocal-fix flow: "fix this phrase with AI" for a REGION of an existing timeline clip, built on top of the v-a repaint client but addressing TIMELINE material rather than a file. `clip.deleteTimeRange` / `clip.insertTimeRange` (M10-h, beta-feedback round — piano-roll "cut a bar"/"insert a bar" moves) — excise/open a CLIP-LOCAL time range in a MIDI clip's notes, over `ProjectStore.deleteTimeRange(clipID:startBeat:lengthBeats:)` / `insertTimeRange(clipID:atBeat:lengthBeats:)` (`Sources/DAWCore/ProjectStore.swift`). Both take CLIP-LOCAL beats — the same space as `clip.setNotes`' note `startBeat` (relative to the clip's own start, never the timeline). `clip.deleteTimeRange` (`clipId`, `startBeat`, `lengthBeats`, all required) removes `[startBeat, startBeat + lengthBeats)` and closes the gap: a note whose onset falls INSIDE the span is REMOVED outright, however far it extends; a note crossing IN from before keeps its head — start unchanged, length reduced by exactly the overlapping portion spliced out; a note at or after the span shifts LEFT by `lengthBeats` (length unchanged). The clip's own `lengthBeats` shrinks by the excised amount, floored up to one beat (`minTimeRangeClipLengthBeats`) or the furthest surviving note's end, whichever is greater — no note is ever left stranded past the clip's end. `clip.insertTimeRange` (`clipId`, `atBeat`, `lengthBeats`, all required) is the inverse: inserts `lengthBeats` of silence at `atBeat`, shifting every note starting at or after it RIGHT by `lengthBeats`; a note starting BEFORE `atBeat` (already sounding across it) keeps BOTH its start and its length untouched — the silence lands after its onset and it simply sustains through the new gap, so no note is ever split. `atBeat` may equal the clip's current length (append silence at the tail). The clip's `lengthBeats` grows by exactly `lengthBeats`. Both route through the shared `requireNotCompMember` guard (a take-comp member throws `clipInTakeGroup`, pointing at `take.setComp`/`take.flatten` — the `clip.quantize`/`clip.humanize` precedent) and reject a non-MIDI target with `notAMIDIClip`; `invalidClipEdit` covers a non-positive `lengthBeats`, a `deleteTimeRange` `startBeat` outside `[0, clipLength)`, or an `insertTimeRange` `atBeat` outside `[0, clipLength]`. ONE `performEdit` per call folds the note edit and the length change into a single undo step. Response: the updated clip. `clip.move` overlap policy + `clip.crossfade` (M11-d, arrange crossfades + overlap correctness) — the SETTLED "no SILENT overlap of ordinary same-track clips" invariant, closing the m10-q audit bug where `ProjectStore.moveClip` did no overlap detection (two overlapped clips summed → ~+6 dB doubling, no crossfade, no error). `moveClip` (`Sources/DAWCore/ProjectStore.swift`) now resolves any overlap the move creates against ordinary same-track clips using trim-window semantics (the shared pure `clipTrimmedToWindow`/`resolveOverlap` helpers): the STATIONARY clip yields the covered region — a covered HEAD trims its leading edge to the moved clip's right edge (audio source offset advances; MIDI notes past the edge drop/truncate — the `trimClip` rules), a covered TAIL or a move landing STRICTLY INSIDE trims its trailing edge to the moved clip's left edge (keep the head — a single trim can't punch a hole), and a fully-covered clip (or a remnant thinner than `minClipLengthBeats`) is REMOVED. Take-group / comp members are exempt (the `requireNotCompMember` precedent). All of it rides the move's single `performEdit` (ONE undo restores everything). `moveClip` now returns `ClipMoveResult {clip, trimmedClipIDs, removedClipIDs}`; the `clip.move` wire response gains ADDITIVE `trimmed:[clipId…]` / `removed:[clipId…]` arrays (never renames the clip fields). A previously-crossfaded clip carries NO special bookkeeping — the crossfade IS just the overlap + complementary fades, so a later move trims against current boundaries and any surviving fades stay as plain per-clip fades. The ONLY sanctioned overlap is `clip.crossfade` (`trackId`, `clipId`, `otherClipId`, `lengthBeats > 0`; ids in either order, earlier-starting clip is the fade-OUT/left side): two AUDIO clips on one track, ADJACENT (right.start == left.end) or already overlapping by ≤ `lengthBeats`. Adjacent → the overlap is CREATED of exactly `lengthBeats`, split symmetrically by extending the left clip's tail and the right clip's head `lengthBeats/2` each into their surrounding source audio (time-aligned so the equal-power sum reconstructs the original signal); if a side lacks material (the left clip's window ends at the source-file end, read via `media`; the right clip's `startOffsetSeconds` < the head extension) it throws `crossfadeNeedsMaterial` NAMING that clip/side. Already overlapping (a legacy pre-fix overlap) → the existing overlap is KEPT and only the fades applied (normalization). Then the left clip gets a fade-OUT and the right a fade-IN, each `FadeCurve.equalPower` (documented to sum to unit power — `Sources/DAWCore/Model.swift`, so NO engine change: two overlapping clips with complementary equal-power fades already render correctly through the sumMixer, i-b), spanning EXACTLY the final overlap; each clip's OTHER fade is preserved (trimmed only on a collision — the overlap fade is authoritative). ONE `performEdit("Crossfade Clips")`; comp members rejected; `crossfadeNotEligible` covers a MIDI clip, a gap, an over-overlap, or same/same-start ids. - **MIDI data model: SETTLED (M3, 2026-07-05)** — a MIDI clip is a regular
  `Clip` carrying an optional `notes: [MIDINote]?` payload instead of an
  `audioFileURL`. The two are MUTUALLY EXCLUSIVE by construction (`Clip.init`:

  model. Deliberately NO per-note MIDI channel field: a clip belongs to one
  instrument track, so channel is a track/routing concern, added later only if
  multi-timbral hosting needs it. Scheduler contract (consumed by M3 (iii)):
  notes are stored CANONICALLY ORDERED (by onset, then pitch, then id), a
  zero-length note is impossible, note bounds are NON-DESTRUCTIVE (clip length
  clips playback but never trims stored notes), and overlapping notes on the

- **Engine lifecycle: SETTLED (m13-a, 2026-07-12)** — Engine lifecycle: once-rendered engines are never surgically detached; teardown-class changes discard and rebuild the engine (m13-a). Measured: 41-track announce rebuild worst 143 ms (≤ 300 ms budget), +0.4 MB RSS over 100 rebuilds, m12-b offline anchors byte-identical.
